# 工作空间（Workspace）

pnpm 内置了对**单一存储库（也称为多包存储库、多项目存储库或单体存储库）的支持**。 你可以创建一个工作空间以将多个项目合并到一个仓库中。

一个工作空间必须在它的根目录有一个 [`pnpm-workspace.yaml`](https://pnpm.io/zh/pnpm-workspace_yaml) 文件。

## pnpm-workspace.yaml

`pnpm-workspace.yaml` 是工作空间的配置文件，位于仓库根目录，用于声明哪些目录是工作空间中的包。

### 基本结构

```yaml
packages:
  # 使用 glob 模式匹配包目录
  - 'packages/*'
  - 'components/**'
  - 'apps/*'
  # 排除某个目录
  - '!**/test/**'
```

### packages 字段

`packages` 是一个 glob 模式数组，用于声明工作空间中包含哪些目录作为子包：

| 模式 | 含义 |
|------|------|
| `'packages/*'` | 匹配 `packages/` 下的所有一级子目录 |
| `'packages/**'` | 匹配 `packages/` 下的所有层级子目录 |
| `'apps/*'` | 匹配 `apps/` 下的所有一级子目录 |
| `'!packages/experimental'` | 排除指定目录，`!` 前缀表示否定 |
| `'!**/test/**'` | 排除所有包含 `test` 的路径 |

::: tip 注意

每个匹配到的目录必须包含自己的 `package.json`，否则 pnpm 会将其视为普通目录而非工作空间子包。

:::

### 常见仓库结构

一个典型的 pnpm 工作空间项目结构如下：

```plaintext
my-monorepo/
├── pnpm-workspace.yaml
├── package.json                # 根 package.json
├── pnpm-lock.yaml
├── packages/
│   ├── core/
│   │   └── package.json        # @my-scope/core 包
│   ├── utils/
│   │   └── package.json        # @my-scope/utils 包
│   └── ui/
│       └── package.json        # @my-scope/ui 包
├── apps/
│   ├── web/
│   │   └── package.json        # Web 应用
│   └── mobile/
│       └── package.json        # 移动端应用
└── docs/
    └── package.json            # 文档站点
```

对应的 `pnpm-workspace.yaml`：

```yaml
packages:
  - 'packages/*'
  - 'apps/*'
  - 'docs'
```

## 子包之间引用: workspace: 协议

在 pnpm 工作空间中，子包之间可以通过 `workspace:` 协议互相引用，而无需将包发布到 npm registry。

### 基本用法

假设工作空间中有两个子包：`@my-scope/core` 和 `@my-scope/web`，`web` 需要依赖 `core`：

```json
{
  "name": "@my-scope/web",
  "dependencies": {
    "@my-scope/core": "workspace:*"
  }
}
```

pnpm 在安装时会自动识别 `workspace:` 协议，将依赖解析为工作空间中的本地包，并建立符号链接。

### workspace: 协议版本范围

| 协议 | 含义 | 说明 |
|------|------|------|
| `workspace:*` | 任意版本 | 最常用，只要工作空间中存在该包即可，不校验版本 |
| `workspace:~` | 补丁版本兼容 | 匹配工作空间中 `~x.y.z` 范围内的版本 |
| `workspace:^` | 次要版本兼容 | 匹配工作空间中 `^x.y.z` 范围内的版本 |
| `workspace:^1.5.0` | 指定范围 | 只匹配工作空间中满足 `^1.5.0` 的版本，不满足则报错 |

::: tip 版本不匹配时会怎样？

如果工作空间中存在 `@my-scope/core@1.0.0`，但 `web` 包声明了 `"@my-scope/core": "workspace:^2.0.0"`，pnpm 会直接报错，不会从远程 registry 回退安装，这样可以确保始终使用本地包。

:::

### 实际示例

基于上面的仓库结构，各子包之间的依赖关系：

```plaintext
@my-scope/core      ← 无内部依赖（基础包）
@my-scope/utils     ← 依赖 @my-scope/core
@my-scope/ui        ← 依赖 @my-scope/core + @my-scope/utils
@my-scope/web       ← 依赖 @my-scope/ui + @my-scope/utils
```

`@my-scope/ui` 的 `package.json`：

```json
{
  "name": "@my-scope/ui",
  "version": "1.0.0",
  "dependencies": {
    "@my-scope/core": "workspace:*",
    "@my-scope/utils": "workspace:*"
  }
}
```

`@my-scope/web` 的 `package.json`：

```json
{
  "name": "@my-scope/web",
  "version": "1.0.0",
  "dependencies": {
    "@my-scope/ui": "workspace:*",
    "@my-scope/utils": "workspace:*"
  }
}
```

### 工作原理

当 pnpm 解析 `workspace:` 依赖时：

1. 扫描 `pnpm-workspace.yaml` 中声明的所有子包
2. 根据包名（`name` 字段）匹配目标子包
3. 在 `node_modules` 中创建**符号链接**指向本地包目录
4. 修改本地包代码后，引用方立即生效，无需重新安装或构建

```plaintext
apps/web/node_modules/
└── @my-scope/
    ├── core  → ../../../../packages/core     # 符号链接
    └── utils → ../../../../packages/utils    # 符号链接
```

### devDependencies 和 peerDependencies

`workspace:` 协议同样适用于 `devDependencies` 和 `peerDependencies`：

```json
{
  "devDependencies": {
    "@my-scope/core": "workspace:*"
  },
  "peerDependencies": {
    "@my-scope/core": "workspace:^1.0.0"
  }
}
```

### 发布时的版本转换

使用 `workspace:` 协议的包在发布到 npm 时，pnpm 会自动将 `workspace:` 协议替换为实际的版本号：

| 开发时 | 发布后 |
|--------|--------|
| `"workspace:*"` | `"1.0.0"`（当前包的实际版本） |
| `"workspace:^"` | `"^1.0.0"` |
| `"workspace:~"` | `"~1.0.0"` |
| `"workspace:^1.5.0"` | `"^1.5.0"`（保持不变） |

::: tip 总结

`workspace:` 协议是 pnpm 工作空间的核心机制，它让子包之间的引用变得简单、安全、高效——无需发布、无需手动链接、版本不匹配时立即报错。

:::

## 安装包

在 pnpm 工作空间中，安装包的命令和普通项目一样，但行为会根据**执行位置**和**参数**有所不同。

### 给指定子包安装依赖

使用 `--filter` 参数给某个子包安装依赖：

```bash
# 给 @my-scope/web 包安装 lodash
pnpm --filter @my-scope/web add lodash

# 给 @my-scope/core 包安装开发依赖
pnpm --filter @my-scope/core add -D typescript
```

`--filter` 支持多种匹配方式：

| 写法 | 含义 |
|------|------|
| `--filter @my-scope/web` | 精确匹配包名 |
| `--filter "./apps/web"` | 匹配指定目录下的包 |
| `--filter "./packages/*"` | 匹配目录下所有子包 |
| `--filter "...@my-scope/core"` | 匹配 @my-scope/core 及其所有依赖包 |

### 给根目录安装依赖

在根目录执行 `pnpm add`，会将依赖安装到根 `package.json`。通常根目录只安装工作空间级别的共享工具：

```bash
# 在根目录安装共享的开发工具
pnpm add -w eslint typescript prettier -D
```

`-w -D` 等价于 `--save-dev --workspace-root`，表示安装到根目录的 `devDependencies`。

::: tip 为什么共享依赖放根目录？

工作空间中所有子包都可以访问根目录的 `devDependencies`，将 ESLint、TypeScript 等共享工具放在根目录可以避免每个子包重复安装，节省磁盘空间和统一管理版本。

:::

### 子包引用其他子包

使用 `workspace:` 协议将一个子包安装为另一个子包的依赖：

```bash
# 给 @my-scope/web 安装工作空间中的 @my-scope/core
pnpm --filter @my-scope/web add @my-scope/core@workspace:*

# 安装为开发依赖
pnpm --filter @my-scope/web add -D @my-scope/utils@workspace:*
```

pnpm 会自动解析工作空间中的本地包，建立符号链接，不会从远程 registry 下载。

### 安装远程包 vs 工作空间包

| 命令 | 行为 |
|------|------|
| `pnpm add lodash` | 从远程 registry 安装 lodash |
| `pnpm add @my-scope/core` | 从工作空间安装本地包（如果存在） |
| `pnpm add @my-scope/core@workspace:*` | 强制从工作空间安装，不存在则报错 |

::: tip 优先级

当工作空间中存在同名包时，pnpm 优先使用本地包。如果希望强制使用远程版本，可以在包名后加上版本号，或使用 `--workspace-root` 参数调整行为。

:::

### 一次性安装所有依赖

在根目录执行 `pnpm install`，pnpm 会解析 `pnpm-workspace.yaml`，一次性安装根目录和所有子包的依赖：

```bash
# 在根目录执行，安装工作空间中所有包的依赖
pnpm install
```

安装过程：

1. 读取根目录 `package.json` 和所有子包的 `package.json`
2. 解析 `workspace:` 协议，建立子包之间的符号链接
3. 从远程 registry 下载所有外部依赖
4. 统一生成一份 `pnpm-lock.yaml` 在根目录

### 常用安装命令汇总

```bash
# 安装所有子包的依赖
pnpm install

# 给指定子包安装外部依赖
pnpm --filter @my-scope/web add axios

# 给指定子包安装工作空间内的子包
pnpm --filter @my-scope/web add @my-scope/core@workspace:*

# 给根目录安装共享开发工具
pnpm add -w eslint prettier typescript -D

# 给多个子包批量安装同一个依赖
pnpm --filter "./packages/*" add lodash

# 卸载指定子包的依赖
pnpm --filter @my-scope/web remove axios
```
