# git rebase

## 作用

用于 **变基（重写提交历史）** 的核心命令，它能将一个分支的提交 “嫁接” 到另一个分支的最新提交上，让提交历史更线性整洁；也能通过交互式模式整理提交历史（合并、修改、删除提交）

与 `git merge` 相比，`rebase` 不会产生额外的合并提交，而是改写提交历史，让提交图看起来像一条直线。

## 概述

```bash
git rebase [-i | --interactive] [<options>] [--exec <cmd>]
	[--onto <newbase> | --keep-base] [<upstream> [<branch>]]
git rebase [-i | --interactive] [<options>] [--exec <cmd>] [--onto <newbase>]
	--root [<branch>]
git rebase (--continue|--skip|--abort|--quit|--edit-todo|--show-current-patch)
```

## 基础使用示例

```bash
# 将 main 分支的提交变基到当前分支
git rebase main

# 语法：git rebase <目标分支> <要变基的分支>
# 相当于先切换到 feature，再执行 git rebase main。
git rebase main feature
```

## 什么是变基？

变基的本质是**找到两个分支的共同祖先，将当前分支的所有提交在目标分支上 “重新应用” 一遍**，最终让当前分支的 “基”（起点）变成目标分支的最新提交，从而避免 merge 产生的 “分叉提交”，让历史更线性。

- **merge**：保留两个分支的历史，生成 merge commit（分叉历史）；
- **rebase**：重写当前分支的提交历史，将其 “接” 在目标分支后面（线性历史）。

**注意: 改变的是当前分支的提交历史, 不会改变目标分支的提交历史**

## 场景 1：基础变基

假设有两个分支：`main` 和 `feature`，它们的提交历史如下：

```text
      A---B---C feature
     /
D---E---F---G main
```

在 `feature` 分支上执行 `git rebase main`，Git 会：

1. 找到共同祖先：找到 `feature` 和 `main` 的共同祖先提交（E）。
2. 生成补丁序列：将 `feature` 分支上从 E 之后的所有提交（A、B、C）转换成一系列 “补丁”（diff 文件）。
3. 切换到新基准：将 `feature` 分支的指针临时移动到 `main` 的最新提交 G。
4. 应用补丁：按顺序将 A/B/C 的补丁逐个应用到 G 上，生成新提交 A'/B'/C'（哈希值不同，但修改内容一致）。
5. 移动分支指针：将 `feature` 分支的指针最终指向 C'，完成变基。

结果变成：

```text
                      A'---B'---C' feature
                     /
D---E---F---G main
```

**原来的 A、B、C 提交会被丢弃**（但可以通过 `reflog` 找回）。这样，`feature` 分支的历史就变基到了 `main` 的最新提交上，看起来像是直接从 G 开始开发的新功能。

## 场景 2: 已提交到远程的分支

假设有两个分支：`main` 和 `feature`，其中 `feature` 已提交到远程分支 `origin/feature`，它们的提交历史如下：

```text
      D---E---F  <-- feature, origin/feature
     /
A---B---C---G  <-- main
```

在 `feature` 分支上执行 `git rebase main`，结果会：

```text
                      D'--E'--F'  <-- feature
                     /
A---B---C---G  <-- main
     \
      D---E---F  <-- origin/feature
```

此时如果直接 `git push`，Git 会拒绝，因为远程分支的历史不是本地的直接祖先。

解决方案: 强制推送 --> `git push --force-with-lease`

**注意: 原则上不要对已经推送到公共仓库、且可能被他人使用的分支进行 rebase。**

## 交互式变基

交互式变基（`git rebase -i`）是 Git 中最强大的历史整理工具之一。它允许**修改、重排、合并、删除提交**，从而让提交历史变得清晰、整洁。

### 语法

- `git rebase -i <base>`

- `<base>`: 可以是分支名、标签、或某个提交的哈希值（如 `HEAD~3`）。Git 会列出从 `<base>` 之后到当前分支最新提交之间的所有提交，供其编辑。

- 示例: `git rebase -i HEAD~3`
  - 执行后会打开一个文本编辑器（通常是 vim 或配置的编辑器），内容类似：

  ```text
  pick a1b2c3d 第一次提交说明
  pick e4f5g6h 第二次提交说明
  pick i7j8k9l 第三次提交说明

  # Rebase 1234567..abcdefg onto 9876543
  # ...
  ```

### 子命令说明

::: tip
可以在交互式列表中**重新排列行的顺序**，Git 会按照新的顺序依次应用提交。
:::

| 命令         | 含义                                                                                     |
| :----------- | :--------------------------------------------------------------------------------------- |
| `pick` (p)   | 使用该提交，不做任何修改。                                                               |
| `reword` (r) | 使用该提交，但在应用时会暂停，让你修改提交信息。                                         |
| `edit` (e)   | 使用该提交，但应用后会暂停，允许你对提交内容进行修改（可以添加、删除文件，或拆分提交）。 |
| `squash` (s) | 将该提交合并到前一个提交中，并把两个提交的日志信息合并，让你编辑新的信息。               |
| `fixup` (f)  | 类似于 `squash`，但丢弃该提交的日志信息，只保留前一个提交的信息。                        |
| `drop` (d)   | 删除该提交。                                                                             |
| `exec` (x)   | 在某个提交之后执行指定的 shell 命令（例如运行测试、格式化代码）。                        |
| `break` (b)  | 在此时停止，让你可以执行其他操作（如 `git commit`），之后用 `--continue` 继续。          |

### 示例: 合并多个提交（Squash）

将多个提交合并成一个逻辑完整的提交

1. 执行 `git rebase -i HEAD~4`
2. 在编辑器中，将后三个提交的命令改为 `squash` 或 `fixup`：
   ```text
   pick abc123 完成功能主体
   squash def456 修复拼写错误
   squash ghi789 添加注释
   squash jkl012 优化性能
   ```
3. 保存退出后，Git 会依次应用，最后弹出编辑器编写合并后的提交信息。

### 示例: 拆分一个提交

将一个提交拆分成多个提交

1. 执行 `git rebase -i HEAD~3`

2. 在编辑器中，将要拆分的提交命令改为 `edit`：

   ```text
   pick abc123 之前的提交
   edit def456 这个大提交包含两个功能
   pick ghi789 之后的提交
   ```

3. 保存退出后，Git 会暂停在 `def456` 应用之后。此时可以：

   ```bash
   git reset HEAD^            # 撤销该提交，但保留工作区修改
   git add -p                 # 交互式选择第一部分改动
   git commit -m "第一部分"
   git add -p                 # 选择第二部分改动
   git commit -m "第二部分"
   git rebase --continue      # 继续应用后续提交
   ```

### 示例: 删除一个提交

直接删除交互式列表中对应的行，或将命令改为 `drop`。

## 注意事项

1. **变基操作会改变提交历史**，因此需要谨慎使用。

2. **不要在公共分支（如 main/dev）上用 rebase**：变基会改写历史，若其他开发者已拉取该分支，会导致他们的本地历史与远程不一致，引发严重协作冲突。

3. **已推送到远程的分支，变基后需强制推送**：若变基的分支已推送到远程，变基后历史已改写，普通 push 会报错，需用 -f 强制推送（仅私有分支可用）：

4. **适用使用场景**:
   - 个人特性分支: 提交多个零散的提交等场景下
   - 使用 `git pull --rebase` 代替默认的 `git pull`

## 选项

### 交互式变基类（精细整理历史）

这类选项用于**进入交互式模式，手动选择、修改、合并、删除提交**，是 git rebase 最强大的功能。

#### `--interactive, -i`: 进入交互式变基模式

- **作用**: 对指定范围内的提交进行交互式编辑
- **示例**: `git rebase -i HEAD~3`

#### `--autostash`: 变基前自动暂存未提交的修改

- **作用**: 若工作区有未提交的修改，变基会失败；用此选项可自动暂存修改，变基完成后自动恢复
- **示例**: `git rebase --autostash main`

### 高级场景类（处理复杂分支操作）

这类选项用于**特殊场景下的变基**，比如精准移动提交段、变基到根提交等。

#### `--onto <新基>`: 精准移动提交段到新基准

- **作用**: 将「从旧基分支切出的提交」直接变基到新基分支，跳过中间分支（最常用的高级选项）
- **示例**: `git rebase --onto main feature sub-feature` （将 `sub-feature` 从 `feature` 切出的提交直接变基到 `main`）

#### `--root`: 变基到根提交（仓库第一个提交）

- **作用**: 将分支的所有提交从根提交开始重新应用（适合重构仓库早期历史）
- **示例**: `git rebase --root`

#### `--force-rebase, -f`: 强制重新变基（即使已变基过）

- **作用**: 若分支之前已变基过，默认不会重复变基；用此选项可强制重新应用所有补丁（谨慎使用！）
- **示例**: `git rebase --force-rebase main`

### 基础操作控制类（变基流程核心）

这类选项用于**控制变基的启动、暂停、继续与终止**，是变基过程中最常用的基础命令。

#### `--continue`: 继续执行暂停的变基

- **作用**: 解决冲突后（`git add` 冲突文件），用此命令继续应用剩余补丁
- **示例**: `git rebase --continue`

#### `--skip`: 跳过当前冲突的提交

- **作用**: 若当前提交的修改不重要，可跳过（谨慎使用！可能导致代码丢失）
- **示例**: `git rebase --skip`

#### `--abort`: 终止变基，恢复到变基前状态

- **作用**: 无论变基到哪一步，都能安全回滚到变基前的分支状态
- **示例**: `git rebase --abort`

#### `--quit`: 退出变基，但保留已应用的修改

- **作用**: 与 `--abort` 不同，`--quit` 会保留变基过程中已成功应用的提交，仅终止后续操作（极少用）
- **示例**: `git rebase --quit`

### 冲突处理类（解决变基冲突）

这类选项用于**辅助处理变基过程中的代码冲突**，降低冲突解决难度。

#### `--ignore-whitespace`: 忽略空白字符差异

- **作用**: 补丁与本地代码的空格、换行、Tab 差异会导致冲突，用此选项可忽略（仅忽略空白，不修改代码）。
- **使用场景**: 忽略空白差异，让 Git 忽略代码格式差异，让冲突更简单。
- **示例**: `git rebase --ignore-whitespace main`

#### `--whitespace=<action>`: 处理空白字符错误

- **作用**: 对空白字符错误（如行尾空格、缩进不一致）执行指定操作
  - `fix`：修复错误（自动删除行尾空格等）；
  - `warn`: 仅警告不修复；
  - `error`: 将空白错误视为冲突。
- **示例**: `git rebase --whitespace=fix main`

#### `--no-verify`: 跳过 pre-rebase 钩子

- **作用**: 若仓库配置了 `pre-rebase` 钩子（变基前的校验脚本），用此选项可跳过校验（谨慎使用！）
- **示例**: `git rebase --no-verify main`

### 历史形态控制类（决定变基后的历史结构）

这类选项用于**控制变基后提交历史的形态**，比如是否保留合并提交、是否强制生成新提交等。

#### `--rebase-merges, -r`: 保留合并提交的结构

- **作用**: 默认变基会丢弃合并提交，将分支线性化；用此选项可保留合并提交的树形结构（适合复杂分支历史）。
- **示例**: `git rebase -r main`

#### `--no-ff`: 强制生成变基提交

- **作用**: 即使变基后可以 “快进”（Fast-forward），也强制生成一个新的变基提交，保留变基痕迹
- **示例**: `git rebase --no-ff main`

#### `--keep-base`: 保持分支的基准不变

- **作用**: 若当前分支的基准（共同祖先）已是目标分支的最新提交，不执行变基（避免无意义的重写历史）
- **示例**: `git rebase --keep-base main`

### 信息展示类（查看变基相关信息）

这类选项用于**查看变基信息，展示变基的变更统计、进度等信息**，帮助用户了解变基影响。

#### `--stat`: 变基完成后显示详细变更统计

- **作用**: 显示变更的文件列表、每个文件的增删行数
- **示例**: `git rebase --stat main`
