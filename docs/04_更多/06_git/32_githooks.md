# githooks

是 Git 提供的**自动化脚本机制**，能在 Git 执行关键操作（如提交、推送、合并）的「前 / 后」触发自定义脚本，实现代码校验、自动化部署、提交规范检查等功能。

## 核心本质

Git Hooks 是存储在仓库 `.git/hooks` 目录下的**可执行脚本**（Shell/Python/Node.js 等），Git 会在特定操作触发时自动执行这些脚本

## 核心特点

- **默认禁用**：Git 初始化仓库时，`.git/hooks` 会生成示例脚本（`.sample` 后缀），需移除后缀并赋予执行权限才生效；

- **本地生效**：默认情况下，`.git/hooks` 不在版本控制范围内（`.gitignore` 自动忽略），仅对当前本地仓库生效；

- **触发时机**：分「前置钩子（pre-）」和「后置钩子（post-）」：
  - 前置钩子（如 `pre-commit`）：操作执行前触发，返回非 0 可终止操作；
  - 后置钩子（如 `post-commit`）：操作执行后触发，仅做通知 / 清理，无法终止操作。

## 钩子类型

Git Hooks 按「执行环境」可明确分为**客户端钩子**（本地仓库触发）和**服务端钩子**（远程仓库触发），核心区别是：客户端钩子在开发者本地执行，服务端钩子仅在远程 Git 服务器（如 GitHub/GitLab/ 自建 Git 服务器）执行。

| 类型                       | 钩子名称             | 触发时机                               | 核心作用                                                 | 能否终止操作 |
| -------------------------- | -------------------- | -------------------------------------- | -------------------------------------------------------- | ------------ |
| **客户端钩子**（本地仓库） | `pre-commit`         | `git commit` 执行后、提交信息编辑前    | 校验待提交代码（lint / 格式化 / 测试）                   | ✅            |
|                            | `prepare-commit-msg` | 提交信息编辑器启动前                   | 自动生成 / 修改提交信息（如补充分支名）                  | ❌            |
|                            | `commit-msg`         | 提交信息编辑后、提交完成前             | 校验提交信息格式（如 Conventional Commits）              | ✅            |
|                            | `post-commit`        | 本地提交完成后                         | 本地通知 / 日志（如提交成功后打印信息）                  | ❌            |
|                            | `pre-push`           | `git push` 执行后、推送到远程前        | 校验待推送代码（单元测试 / 分支命名）                    | ✅            |
|                            | `pre-rebase`         | `git rebase` 执行前                    | 阻止危险的变基操作（如对已推送分支变基）                 | ✅            |
|                            | `post-rewrite`       | 改写提交历史后（rebase/amend 等）      | 记录历史改写操作                                         | ❌            |
|                            | `post-checkout`      | `git checkout`/`git switch` 后         | 切换分支后自动执行（如安装依赖 / 更新配置）              | ❌            |
|                            | `post-merge`         | `git merge` 完成后                     | 合并分支后执行（如更新子模块 / 重启服务）                | ❌            |
|                            | `pre-auto-gc`        | Git 自动垃圾回收（gc）前               | 阻止不必要的 gc，或执行前置清理                          | ✅            |
| **服务端钩子**（远程仓库） | `pre-receive`        | 远程仓库接收推送（`git push`）前       | 全局校验（分支保护 / 权限检查 / 代码规范），拒绝违规推送 | ✅            |
|                            | `update`             | 远程仓库处理每个分支的推送前           | 按分支粒度校验（如禁止推送到 master 分支）               | ✅            |
|                            | `post-receive`       | 远程仓库接收推送完成后                 | 服务端自动化（部署生产环境 / 触发 CI / 发送通知）        | ❌            |
|                            | `post-update`        | 远程仓库更新引用（分支 / 标签）后      | 轻量化通知（如更新仓库索引）                             | ❌            |
|                            | `pre-receive-pack`   | 远程仓库接收 `git fetch`/`git pull` 前 | 限制拉取权限（极少用）                                   | ✅            |

### 客户端钩子

- **执行位置**：开发者的本地电脑仓库（`.git/hooks` 目录）；

- **管控范围**：仅约束当前开发者的本地操作，无法影响团队其他成员；

- **典型场景**：

  - 提交前强制代码 lint（`pre-commit`）；
  - 校验提交信息格式（`commit-msg`）；
  - 推送前执行单元测试（`pre-push`）；

  

- **特点**：可被开发者用 `--no-verify` 跳过（紧急情况）。

### 服务端钩子（远程）

- **执行位置**：Git 服务器的仓库（如 GitHub/GitLab 服务器、自建 Git 服务器的仓库 `.git/hooks` 目录）；

- **管控范围**：约束所有开发者的推送操作，是团队代码规范的「最后一道防线」；

- **典型场景**：

  - 禁止直接推送到 master/main 分支（`pre-receive`/`update`）；
  - 校验推送的代码是否通过 CI 检查（`pre-receive`）；
  - 推送成功后自动部署生产环境（`post-receive`）；

  

- **特点**：无法被客户端跳过，是强制约束。

## 团队共享 Hooks

默认情况下，`.git/hooks` 不在版本控制范围内，需通过以下方式让团队共享钩子：

### 手动同步

1. 在项目根目录创建 `hooks` 目录，存放共享的钩子脚本；
2. 添加同步脚本, 手动执行脚本同步钩子到 `.git/hooks`  目录

### 使用 `core.hooksPath` 配置

Git 支持自定义钩子目录，无需复制脚本，直接指定共享目录。

通过 Git 配置 `core.hooksPath` 指向该目录（或通过脚本自动配置）

### 通过构建脚本或包管理器自动复制钩子

在项目的构建脚本中添加一个钩子，将版本库中保存的钩子文件复制到每个成员的 `.git/hooks` 目录。

**示例**（npm script）：

```json
{
  "scripts": {
    "postinstall": "cp -r .githooks/* .git/hooks/"
  }
}
```

### 使用专用的钩子管理工具

1. 使用常用的 `simple-git-hooks` 钩子
2. 在 `package.json` 增加 `"postinstall": "simple-git-hooks"` 脚本, 借助 npm钩子在安装包之后自动执行命令
