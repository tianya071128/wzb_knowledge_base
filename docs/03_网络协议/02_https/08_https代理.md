# HTTPS 代理

HTTPS 代理的核心问题是：**代理服务器如何转发加密的 HTTPS 流量？**

HTTP 代理可以直接读取、修改请求报文再转发，但 HTTPS 运行在 TLS 之上，数据是加密的，代理无法"看懂"内容。解决方案就是 **CONNECT 隧道代理**。

## CONNECT 隧道代理

CONNECT 方法是 HTTP/1.1 中专门为实现隧道代理设计的。它的基本思想是：**代理不解析 HTTPS 报文，只负责在客户端和目标服务器之间建立一条"透明的 TCP 隧道"，后续所有数据原封不动地双向转发。**

整个过程分为两个阶段：

1. **建隧道**：客户端通过 CONNECT 方法告诉代理"我要连接哪个目标服务器"，代理建立到目标的 TCP 连接后回复 `200 Connection Established`。
2. **传数据**：隧道建立后，客户端与目标服务器之间的所有通信（TLS 握手、加密数据）都通过这条隧道透明传输，代理完全看不到内容。

::: tip 关键点

CONNECT 隧道建立后，代理退化为一个纯粹的 **TCP 转发器**，不参与任何应用层协议的处理。客户端与目标服务器之间的 TLS 握手、证书验证、加密通信完全端到端完成，代理"看不见"也"摸不着"。

:::

## 客户端 → 代理 → 目标服务器

### 完整流程

```
客户端                          代理服务器                    目标服务器
  |                               |                            |
  |  1. CONNECT target:443 HTTP/1.1                           |
  |------------------------------>|                            |
  |                               |  2. TCP 三次握手            |
  |                               |--------------------------->|
  |                               |<---------------------------|
  |  3. HTTP/1.1 200 Connection Established                   |
  |<------------------------------|                            |
  |                               |                            |
  |  ========= TCP 隧道建立完成 =========                       |
  |                               |                            |
  |  4. TLS ClientHello (加密)    |                            |
  |------------------------------>|  5. 原样转发 TLS ClientHello|
  |                               |--------------------------->|
  |                               |                            |
  |                               |  6. TLS ServerHello + 证书  |
  |                               |<---------------------------|
  |  7. TLS ServerHello + 证书 (原样转发)                      |
  |<------------------------------|                            |
  |                               |                            |
  |  ... TLS 握手完成 ...         |                            |
  |                               |                            |
  |  8. 加密的 HTTP 请求           |                            |
  |------------------------------>|  9. 原样转发                |
  |                               |--------------------------->|
  |                               |                            |
  |                               |  10. 加密的 HTTP 响应       |
  |                               |<---------------------------|
  |  11. 原样转发加密响应          |                            |
  |<------------------------------|                            |
```

### 各步骤详解

#### 阶段一：建立隧道

1. **客户端发送 CONNECT 请求**：客户端向代理发送一个特殊的 HTTP 请求，使用 `CONNECT` 方法，目标地址写的是 `host:port`（而不是完整的 URL）：

```http
CONNECT www.example.com:443 HTTP/1.1
Host: www.example.com:443
Proxy-Authorization: Basic dXNlcjpwYXNz
```

::: tip 注意

CONNECT 请求中的请求行只包含 `host:port`，不包含 scheme（`https://`），因为代理不需要知道协议类型，它只需要知道目标地址和端口。

:::

2. **代理建立到目标服务器的 TCP 连接**：代理解析 CONNECT 请求中的目标地址，与目标服务器（`www.example.com:443`）完成 TCP 三次握手。

::: tip 代理 → 目标服务器：纯 TCP 连接建立

这一阶段代理与目标服务器之间**只交换 TCP 控制报文，不传输任何应用层数据**：

```
代理                                     目标服务器
  |                                         |
  |  SYN (seq=x)                            |
  |---------------------------------------->|
  |                                         |
  |  SYN-ACK (seq=y, ack=x+1)              |
  |<----------------------------------------|
  |                                         |
  |  ACK (seq=x+1, ack=y+1)                |
  |---------------------------------------->|
  |                                         |
  |  ==== TCP 连接建立完成，无数据传输 ====   |
  |                                         |
```

- **SYN**：代理向目标服务器发起连接请求
- **SYN-ACK**：目标服务器确认并回复
- **ACK**：代理确认，TCP 连接建立

三次握手的目的是在代理和目标服务器之间建立一条可靠的传输通道，但此时通道是"空"的——没有任何 HTTP 请求、TLS 握手或其他应用层数据通过。代理只是拿到了一个已连接的 TCP socket，等待后续隧道中使用。

目标服务器在此时**完全不知道**即将通过这条连接传输的是 HTTPS 流量，它只看到了一条普通的 TCP 连接。

**但这不影响目标服务器后续识别应用协议——它依赖的是端口号，而不是连接建立时的任何协商。** 当隧道建立后，客户端发出的第一个数据包（TLS ClientHello）到达目标服务器时，服务器根据本地监听的端口来判断协议：

| 端口 | 预期协议 | 第一个数据包 |
|------|---------|------------|
| 443 | TLS（HTTPS） | 期望收到 TLS ClientHello |
| 80 | HTTP | 期望收到 HTTP 请求行（GET/POST 等） |
| 22 | SSH | 期望收到 SSH 协议标识 |
| 587 | SMTP（STARTTLS） | 期望收到 SMTP 握手 |

也就是说，CONNECT 请求中 `www.example.com:443` 的 **443** 已经告诉目标服务器："这条连接上跑的是 TLS"。代理不需要、也没有能力传递任何协议信息——协议识别完全由端口号决定。这就是为什么 CONNECT 请求只需要 `host:port`，不需要 scheme 或其他协议标识。

:::

3. **代理回复 200**：TCP 连接建立成功后，代理向客户端返回 `200 Connection Established`，表示隧道已就绪：

```http
HTTP/1.1 200 Connection Established
```

如果代理无法连接目标（被拒绝、超时、目标不存在等），则返回相应的错误码（如 `403`、`502`、`504`）。

#### 阶段二：通过隧道传输数据

 **隧道建立后**，客户端直接与目标服务器进行 TLS 握手和加密通信。代理对数据完全透明：

- 客户端发出的 TLS ClientHello → 代理原样转发给目标服务器
- 目标服务器返回的证书、ServerHello → 代理原样转发给客户端
- TLS 握手完成后，客户端发送的加密 HTTP 请求 → 代理原样转发
- 目标服务器返回的加密 HTTP 响应 → 代理原样转发

**代理在整个过程中只做一个"搬运工"，不解密、不解析、不修改任何数据。**

### 为什么代理看不到 HTTPS 内容？

```
客户端                              代理                          目标服务器
  |                                  |                               |
  |  <-- TLS 加密通道（端到端） -->                                    |
  |  [明文 HTTP] → [TLS 加密] → [TCP 转发] → [TCP 转发] → [TLS 解密] → [明文 HTTP]
  |                                  |                               |
```

TLS 握手是客户端与目标服务器之间端到端完成的，会话密钥只有两端拥有。代理虽然持有 TCP 连接，但没有密钥，无法解密隧道中的数据。这也是 HTTPS 安全性的体现——即使经过不可信的中间节点，数据仍然是保密和完整的。

## 多层代理

在实际场景中，客户端和目标服务器之间可能存在多个代理，形成**代理链（Proxy Chain）**。每一层代理都需要通过 CONNECT 方法逐层建立隧道。

### 两层代理示例

```
客户端          代理 A（外层）       代理 B（内层）        目标服务器
  |                |                   |                    |
  | 1. CONNECT proxyB:port             |                    |
  |--------------->|                   |                    |
  |                | 2. CONNECT target:443                  |
  |                |------------------>|                    |
  |                |                   | 3. TCP 连接目标     |
  |                |                   |------------------->|
  |                |                   |<-------------------|
  |                | 4. 200 OK         |                    |
  |                |<------------------|                    |
  | 5. 200 OK      |                   |                    |
  |<---------------|                   |                    |
  |                |                   |                    |
  | ========== 双层隧道建立完成 ==========                   |
  |                |                   |                    |
  | 6. TLS ClientHello (加密)          |                    |
  |--------------->| 7. 原样转发        |                    |
  |                |------------------>| 8. 原样转发         |
  |                |                   |------------------->|
  |                |                   |                    |
  |                |                   | 9. TLS ServerHello  |
  |                |                   |<-------------------|
  |                | 10. 原样转发       |                    |
  |                |<------------------|                    |
  | 11. 原样转发    |                   |                    |
  |<---------------|                   |                    |
```

### 多层代理的建立过程

以两层代理（客户端 → 代理 A → 代理 B → 目标服务器）为例：

1. **客户端 → 代理 A**：客户端向代理 A 发送 `CONNECT proxyB:port`，要求代理 A 连接到代理 B。代理 A 与代理 B 建立 TCP 连接后，返回 `200`，第一层隧道建立。

2. **客户端 → 代理 A → 代理 B**：客户端通过第一层隧道（代理 A 转发）向代理 B 发送 `CONNECT target:443`。代理 B 与目标服务器建立 TCP 连接后返回 `200`，第二层隧道建立。

3. **端到端通信**：双层隧道建立后，客户端与目标服务器之间的 TLS 握手和加密数据经过代理 A 和代理 B 逐层原样转发，任何一层代理都无法解密内容。

### N 层代理的通用模式

```
客户端 → 代理1 → 代理2 → ... → 代理N → 目标服务器
```

建立过程是从外到内逐层 CONNECT：

1. 客户端 CONNECT 代理1 → 连接到代理2
2. 客户端通过代理1 CONNECT 代理2 → 连接到代理3
3. ...
4. 客户端通过代理1~N-1 CONNECT 代理N → 连接到目标服务器

每一层 CONNECT 都在上一层的隧道内部建立新的隧道，最终形成一条从客户端到目标服务器的多层嵌套隧道。TLS 握手仍然是客户端与目标服务器之间端到端完成的，所有中间代理都只是透明的 TCP 转发器。

::: warning 性能影响

每增加一层代理，都会增加：

- **连接延迟**：多一层 TCP 握手 + CONNECT 请求往返
- **传输延迟**：数据多经过一个网络节点的转发
- **故障点**：任何一层代理断开，整条链路中断

:::

## 与 HTTP 代理的区别

| 特性 | HTTP 代理 | HTTPS 代理（CONNECT 隧道） |
|------|----------|--------------------------|
| 请求方法 | GET、POST 等 | CONNECT |
| 代理可见性 | 可读取、修改请求/响应内容 | 完全不可见，只能看到加密数据 |
| 代理参与度 | 解析 HTTP 报文，可能修改 Header | 仅做 TCP 层转发 |
| TLS 端点 | 客户端 ↔ 代理（MITM 场景）或客户端 ↔ 目标 | 客户端 ↔ 目标服务器（端到端） |
| 安全性 | 代理可篡改数据 | 数据完整性和机密性由 TLS 保证 |

::: tip 中间人攻击（MITM）

有些企业代理或抓包工具（如 Charles、Fiddler）会"破解" HTTPS 隧道：代理收到 CONNECT 后不直接转发，而是自己与目标服务器建立 TLS 连接，再与客户端建立另一个 TLS 连接（使用自签证书），从而"拆开"隧道，分别加解密两端数据。这本质上已经不是代理，而是中间人攻击。前提是客户端必须信任代理的 CA 证书。

:::
