## HTTP 管道化(流水线)为什么基本上不支持?

[管道化](/http/connection.html#管道化连接-pipeline)，难以现有网络中有老旧的软件与现代版本的软件同时存在，难以保证响应的顺序，已被证明很难在现有的网络中实现。

[具体可参考：知乎问题](https://www.zhihu.com/question/340651010)

## HTTP 为什么被称为文本协议?

区分二进制协议与文本协议：指的是协议本身的数据格式，而不是负载（payload）的格式。

HTTP/1 中，请求行、头、body 里的分隔符，都是 ASCII 码。

HTTP/2，是二进制帧，用字节、位来表示信息，没有 ASCII 码

[stackoverflow](https://stackoverflow.com/questions/2645009/binary-protocols-v-text-protocols)

## HTTP 协议如何判断响应体(请求体)的结束？

[HTTP 协议](https://httpwg.org/specs/rfc7230.html#rfc.section.3.3.3) 中列举了消息体的长度情况，大致可分为：

- 已知的消息体长度，此时通过头部字段：[`Content-Type`](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Headers/Content-Type) 判断，当读取消息体时，会读取指定的长度，超时部分直接丢弃掉。

- 不确定的消息体长度，一般采用[分块传输](/http/bigFile.html#分块传输)方式，此时消息体就要符合规范。

![img](/img/http_01.jpg)

对于不符合规范的请求(响应)，那么可能：

- 响应 [`400 Bad Request`](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Status/400)

- 判断消息正文长度为零

- ...

## 虚拟主机

虚拟主机是**Web 服务器（Nginx/Apache 等）的核心基础技术**，核心能力是：**在单台物理 / 云服务器、单个公网 IP、同一个监听端口（80/443）上，托管多个相互独立的网站 / 域名 / 应用**。

### 实现原理

虚拟主机的本质，是**Web 服务器在收到请求后，通过 “标识字段” 匹配对应的站点配置，将请求转发到对应站点的资源中**。

完整请求链路（全局视角）:

```plaintext
用户输入https://a.com → 浏览器DNS解析a.com到服务器IP 1.2.3.4
→ 浏览器向1.2.3.4:443发起TCP连接
→ TLS握手（浏览器在Client Hello中携带SNI域名）
→ Web服务器根据SNI匹配对应虚拟主机，返回该域名的SSL证书
→ TLS握手完成，加密通道建立
→ 浏览器发送HTTP请求，携带Host: a.com请求头
→ Web服务器再次校验Host头，匹配对应虚拟主机的配置
→ 读取该站点根目录的资源，返回给用户
```

* **HTTP 虚拟主机**: **核心是Host请求头**

  * 浏览器发送 HTTP 请求，请求头中携带`Host: a.com`，明确告知服务器要访问的域名；
  * Web 服务器（Nginx/Apache）收到请求后，提取`Host`头的域名，和自身配置的所有虚拟主机的`server_name`（站点域名）做匹配；
  * 匹配到对应虚拟主机后，执行该站点的配置规则，读取对应根目录的网站资源，返回给用户；
  * 若匹配失败，会走服务器的**默认虚拟主机**（通常是服务器的默认欢迎页、第一个加载的站点）
* **HTTPS 虚拟主机**: **核心是`SNI` TLS 扩展**
  * HTTPS 和 HTTP 的核心差异是：**TLS 握手先于 HTTP 请求发生**。
  * 浏览器先和服务器完成 TLS 握手（验证证书、协商加密密钥），之后才能发送加密的 HTTP 请求；
  * **SNI（Server Name Indication）：HTTPS 虚拟主机的解决方案**
    * SNI 是 TLS 协议的扩展字段，浏览器在**TLS 握手的第一个报文 Client Hello 中，就会携带要访问的目标域名（Server Name 字段）**
    * Web 服务器收到后，根据 SNI 的域名，匹配对应的虚拟主机，提取该站点配置的 SSL 证书，返回给浏览器
    * 浏览器验证证书合法后，完成 TLS 握手，建立加密通道
    * **后续加密的 HTTP 请求中，依然会携带`Host`头，服务器二次校验匹配虚拟主机，返回对应站点资源**

### 使用端口区分程序

在服务器中可以**通过端口区分不同程序**，无需配备多个域名。但一般仅适合**非对外、低体验要求**的场景

| 维度                | 使用端口区分程序                                             | 使用域名区分程序（虚拟主机）                                 |
| ------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **核心原理**        | 同一 IP，不同端口（如 `1.2.3.4:8080`/`8081`），操作系统通过端口转发请求到对应程序 | 同一 IP + 同一端口（80/443），Web 服务器（Nginx/Apache）通过`Host`头 / SNI 匹配域名，转发到对应程序 |
| **访问形式**        | 必须带端口号（如 `http://xxx.com:8080`），用户需记忆端口     | 无需带端口（默认 80/443），仅需记忆域名（如 `a.com`/`b.com`） |
| **用户体验**        | 差（端口号难记、易输错，非标准端口易被浏览器 / 防火墙拦截）  | 优（符合用户访问习惯，和公网网站访问方式一致）               |
| **技术依赖**        | 仅依赖操作系统端口监听，无需 Web 服务器（可直接访问程序端口） | 依赖 Web 服务器（Nginx/Apache）的虚拟主机配置，HTTPS 需依赖 SNI |
| **端口占用 / 管理** | 需规划大量端口（如 8080/8081/9000 等），易冲突，需频繁排查端口占用 | 仅需占用 80/443 端口，所有程序通过 Web 服务器转发，端口管理极简 |
| **安全性**          | 直接暴露程序端口，若程序有漏洞易被攻击；非标准端口易被防火墙拦截 | 仅暴露 80/443 端口，程序端口对内隐藏，Web 服务器可做一层防护（限流、过滤） |
| **HTTPS 适配**      | 每个端口需单独配置 SSL 证书，证书绑定端口无意义（证书绑定域名），易出现证书错误 | 单 IP+80/443 端口可配置多个域名的 SSL 证书（依赖 SNI），证书匹配域名，无安全提示 |
| **运维成本**        | 高（端口规划、冲突排查、外网放行多个端口）                   | 低（仅维护 80/443 端口，域名解析 + Web 服务器配置即可）      |
| **扩展性**          | 端口数量有限（0-65535），大量程序部署易耗尽；跨域配置复杂    | 域名可无限新增，仅需新增 Web 服务器虚拟主机配置，扩展性无上限 |
| **适用场景**        | 本地开发、内部测试、非对外服务（如后台管理系统）             | 公网网站、用户访问的产品、企业官网、多租户平台               |
