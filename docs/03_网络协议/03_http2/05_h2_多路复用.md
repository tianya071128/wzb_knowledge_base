# 多路复用(Multiplexing)

多路复用是 HTTP/2 最核心的特性之一，它允许在**单个 TCP 连接**上同时传输多个请求和响应，彻底解决了 HTTP/1.1 的"队头阻塞"问题，大幅提升了连接利用率和页面加载性能。

## 为什么需要多路复用？

### HTTP/1.1 的问题

在 HTTP/1.1 中，虽然可以使用持久连接（Keep-Alive）复用 TCP 连接，但存在严重的限制：

#### 1. 串行处理（队头阻塞）

HTTP/1.1 采用**请求-响应严格按序**的模型：

```
客户端                    服务器
  |                          |
  |-------- 请求1 --------->|
  |                          | (处理中...)
  |<------- 响应1 ----------|
  |                          |
  |-------- 请求2 --------->|  ← 必须等待响应1完成
  |                          |
```

**问题**：如果请求1的处理时间很长（比如查询数据库），请求2、请求3...都必须等待，即使它们是独立的、互不依赖的资源。这就是著名的"**队头阻塞**"（Head-of-Line Blocking）问题。

#### 2. 多连接方案的开销

为了绕过队头阻塞，浏览器采用了**多连接并发**的方案：

- 浏览器对每个域名建立 **6 个 TCP 连接**（Chrome、Firefox 的默认值）
- 每个连接上串行发送请求

**代价**：
- **连接建立开销**：每个连接需要 TCP 三次握手 + TLS 握手（HTTPS）
- **资源消耗**：服务器需要维护大量连接状态
- **拥塞控制冲突**：多个连接独立进行拥塞控制，可能相互竞争带宽

#### 3. 无法充分利用连接

即使使用 HTTP/1.1 的管道化（Pipelining），也存在问题：
- **响应必须按序返回**：即使服务器先处理完请求2，也必须等请求1的响应发送完
- **实现复杂**：需要处理复杂的边界情况
- **支持有限**：很多浏览器和服务器默认禁用管道化

### HTTP/2 的解决方案

HTTP/2 通过**多路复用**彻底解决了这些问题：

```
客户端                    服务器
  |                          |
  |--- 帧(流1) ------------>|
  |--- 帧(流3) ------------>|  同时发送
  |--- 帧(流1) ------------>|
  |--- 帧(流5) ------------>|
  |                          |
  |<-- 帧(流3) -------------|  乱序返回
  |<-- 帧(流1) -------------|
  |<-- 帧(流5) -------------|
  |<-- 帧(流1) -------------|
```

**核心优势**：
- **一个连接足够**：不再需要建立多个 TCP 连接
- **真正的并发**：多个请求/响应可以交错传输
- **无队头阻塞**：每个流独立处理，互不干扰
- **高效利用连接**：连接始终处于忙碌状态，没有空闲等待

## 多路复用的实现原理

HTTP/2 的多路复用建立在三个核心概念之上：**流（Stream）**、**帧（Frame）** 和 **流 ID（Stream Identifier）**。

### 核心机制

#### 1. 消息分片为帧

HTTP/2 将每个请求/响应消息拆分为多个**二进制帧**：

```
HTTP/1.1 请求：
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Browser

↓ HTTP/2 分片

HEADERS 帧（流ID=1）: 包含 :method=GET, :path=/index.html, ...
DATA 帧（流ID=1）    : 请求体数据（如果有）
```

#### 2. 帧携带流 ID

每个帧的头部都包含一个 **31 位的流标识符**：

```
+-----------------------------------------------+
|                 Length (24)                   |
+-----------------------------------------------+
|   Type (8)    |   Flags (8)                   |
+-----------------------------------------------+
|R|         Stream Identifier (31)              |
+-----------------------------------------------+
|                 Frame Payload                 |
+-----------------------------------------------+
```

**流 ID 的作用**：
- 接收端根据流 ID 将帧"归类"到对应的流中
- 不同流的帧可以在连接上交错传输
- 同一流内的帧按序组装成完整消息

#### 3. 流是虚拟的逻辑通道

**流（Stream）是 HTTP/2 抽象出来的逻辑概念**，在物理传输层面并不存在：

```
TCP 连接上实际传输的是：
[帧(流1)] [帧(流3)] [帧(流1)] [帧(流5)] [帧(流3)] ...

接收端根据流 ID 重组：
流1: [帧1] [帧3] → 完整的请求/响应1
流3: [帧2] [帧5] → 完整的请求/响应3
流5: [帧4]      → 完整的请求/响应5
```

::: tip 为什么说"流"是虚拟的？

从 TCP 栈的角度看，连接上只有一个个独立的帧，每个帧携带一个流 ID。**流本身并不作为一个物理实体存在**，它只是 HTTP/2 协议层的逻辑抽象。

接收端收到帧后，根据流 ID 将帧"归类"到对应的流中，然后按顺序组装成完整的 HTTP 消息。

:::

### 完整工作流程

以浏览器请求一个网页为例：

```
1. 建立连接
   客户端 ←→ 服务器：建立 TCP + TLS 连接
   双方交换 SETTINGS 帧

2. 并发发送多个请求
   客户端 → 服务器：
     HEADERS 帧（流ID=1，请求 /index.html）
     HEADERS 帧（流ID=3，请求 /style.css）
     HEADERS 帧（流ID=5，请求 /app.js）

3. 服务器并发处理并返回
   服务器 → 客户端：
     HEADERS 帧（流ID=1，200 OK）
     HEADERS 帧（流ID=3，200 OK）
     DATA 帧（流ID=1，HTML 内容片段1）
     DATA 帧（流ID=3，CSS 内容）
     DATA 帧（流ID=1，HTML 内容片段2）
     HEADERS 帧（流ID=5，200 OK）
     DATA 帧（流ID=5，JS 内容）
     DATA 帧（流ID=1，END_STREAM）

4. 客户端重组消息
   流1: 完整的 HTML 响应
   流3: 完整的 CSS 响应
   流5: 完整的 JS 响应
```

## 多路复用的关键特性

### 1. 请求并发

**在单个 TCP 连接上，客户端可以同时发起多个请求**，无需等待前一个请求完成：

```javascript
// HTTP/1.1：必须等待或建立多个连接
fetch('/api/user')     // 连接1
fetch('/api/posts')    // 连接2（需要额外连接）
fetch('/api/comments') // 连接3（需要额外连接）

// HTTP/2：单个连接上并发发送
// 所有请求在同一个连接上，通过不同的流 ID 区分
fetch('/api/user')     // 流1
fetch('/api/posts')    // 流3
fetch('/api/comments') // 流5
```

### 2. 响应乱序

**响应可以按处理完成的顺序返回，不必按请求顺序**：

```
请求顺序：流1 → 流3 → 流5
响应顺序：流3 → 流1 → 流5  （完全合法）
```

这解决了 HTTP/1.1 管道化的核心问题：即使请求1还在处理，请求3的响应可以先返回。

### 3. 流独立

**每个流都是独立的，互不干扰**：

- 每个流有独立的状态机（idle → open → half-closed → closed）
- 每个流有独立的流量控制窗口
- 每个流可以设置独立的优先级
- 一个流的阻塞不影响其他流

### 4. 单连接复用

**整个页面只需要一个 TCP 连接**：

| 对比项         | HTTP/1.1                    | HTTP/2              |
| :------------- | :-------------------------- | :------------------ |
| 连接数         | 每域名 6 个                 | 每域名 1 个         |
| TCP 握手开销   | 6 × (TCP + TLS)             | 1 × (TCP + TLS)     |
| 拥塞控制       | 6 个独立拥塞窗口，相互竞争  | 1 个拥塞窗口，统一  |
| 队头阻塞       | 存在                        | 不存在              |

## 多路复用的限制与控制

虽然多路复用允许并发传输，但 HTTP/2 也提供了必要的控制机制，防止资源耗尽。

### 1. 并发流数量限制

通过 `SETTINGS_MAX_CONCURRENT_STREAMS` 参数限制：

```
SETTINGS 帧：
  SETTINGS_MAX_CONCURRENT_STREAMS = 100

含义：对端最多可以同时打开 100 个流
```

**作用**：
- 防止客户端创建过多流消耗服务器资源
- 防止 DoS 攻击
- 确保公平调度

### 2. 流量控制

每个流和整个连接都有**流量控制窗口**：

```
流级窗口：限制单个流可以发送的数据量
连接级窗口：限制整个连接可以发送的总数据量

发送 DATA 帧前，必须确保：
  流级剩余窗口 > 0  且  连接级剩余窗口 > 0
```

**作用**：
- 防止发送方淹没接收方
- 允许接收方控制数据流速

### 3. 流优先级

客户端可以为流设置**优先级**，指导服务器分配带宽：

```
流1（HTML）：权重 256，优先级最高
流3（CSS）：权重 192，次高
流5（图片）：权重 64，较低
```

**作用**：
- 关键资源优先传输
- 优化页面加载体验

## 多路复用 vs HTTP/1.1 对比

### 性能对比示例

假设加载一个包含 6 个资源的页面：

**HTTP/1.1（6 个连接）**：
```
时间线：
连接1: [请求1]---[响应1]
连接2: [请求2]---[响应2]
连接3: [请求3]---[响应3]
连接4: [请求4]---[响应4]
连接5: [请求5]---[响应5]
连接6: [请求6]---[响应6]

开销：6 × (TCP握手 + TLS握手) = 6 RTT（TCP）+ 6 RTT（TLS）
```

**HTTP/2（1 个连接）**：

```
时间线：
连接: [请求1][请求2][请求3][请求4][请求5][请求6]
      [响应2][响应1][响应3][响应5][响应4][响应6]

开销：1 × (TCP握手 + TLS握手) = 1 RTT（TCP）+ 1 RTT（TLS）
```

### 详细对比表

| 特性             | HTTP/1.1                              | HTTP/2 多路复用                       |
| :--------------- | :------------------------------------ | :------------------------------------ |
| **连接数**       | 每域名 6 个                           | 每域名 1 个                           |
| **请求方式**     | 每连接串行                            | 单连接并发                            |
| **队头阻塞**     | 存在（应用层）                        | 不存在（应用层）                      |
| **响应顺序**     | 必须按请求顺序                        | 可以乱序                              |
| **连接开销**     | 高（多次握手）                        | 低（一次握手）                        |
| **拥塞控制**     | 多个独立窗口，可能竞争                | 单一窗口，统一管理                    |
| **资源消耗**     | 高（多连接状态）                      | 低（单连接状态）                      |
| **优先级控制**   | 无                                    | 支持（流优先级）                      |
| **流量控制**     | 依赖 TCP                              | 应用层 + TCP 双重控制                 |



## 注意事项与最佳实践

### 1. 并非所有场景都适合多路复用

**适合的场景**：
- 加载大量小资源（图片、CSS、JS）
- API 批量请求
- 长连接场景（WebSocket 替代）

**不适合的场景**：
- 单个大文件下载（多连接可能更快）
- 对带宽要求极高的场景

### 2. TCP 队头阻塞仍然存在

HTTP/2 解决了**应用层的队头阻塞**，但 TCP 层的队头阻塞仍然存在：

```
如果 TCP 包丢失：
- 所有流都会被阻塞（等待重传）
- 这是 HTTP/3（QUIC）要解决的问题
```

### 3. 合理设置并发流数量

```nginx
# Nginx 配置示例
http2_max_concurrent_streams 128;
```

**建议**：
- 服务器：100-200（根据硬件能力）
- 客户端：浏览器通常自动管理

### 4. 利用流优先级

```javascript
// 现代浏览器会自动设置优先级
// HTML > CSS > JS > 图片
// 开发者也可以通过 Fetch API 提示优先级
fetch('/critical.css', { priority: 'high' });
fetch('/analytics.js', { priority: 'low' });
```

## 总结

HTTP/2 的多路复用通过**流、帧、流 ID** 三个核心概念，实现了在单个 TCP 连接上并发传输多个请求/响应：

| 核心要点           | 说明                                                |
| :----------------- | :-------------------------------------------------- |
| **单连接并发**     | 一个 TCP 连接足够，无需多连接                       |
| **无队头阻塞**     | 每个流独立，互不干扰                                |
| **响应乱序**       | 响应可以按处理完成顺序返回                          |
| **流是虚拟的**     | 物理上只有帧，流是逻辑抽象                          |
| **有控制机制**     | 并发流限制、流量控制、优先级调度                    |
| **性能提升显著**   | 减少连接开销、提高连接利用率、加速页面加载          |

多路复用是 HTTP/2 性能提升的基石，配合头部压缩、服务器推送等特性，使得 HTTP/2 成为现代 Web 的标准协议。
