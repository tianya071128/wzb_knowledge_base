# 流(Stream)

流（Stream）是 HTTP/2 中最核心的概念，它是**二进制帧的双向传输序列**，同一个请求-响应往返的所有帧都会分配一个唯一的流 ID（Stream ID）。

## 为什么需要流？

在 HTTP/1.1 中，虽然可以使用持久连接（Keep-Alive）发送多个请求，但这些请求必须**严格按顺序发送和接收**，也就是常说的"队头阻塞"（Head-of-Line Blocking）问题。如果第一个请求处理很慢，后面的请求就必须等待。

HTTP/2 通过"流"的概念解决了这个问题：

- **多个请求可以并发**：在一个 TCP 连接上同时传输多个请求和响应
- **没有顺序依赖**：每个请求-响应都是独立的流，互不干扰
- **高效利用连接**：避免了队头阻塞，大幅提升性能

## 流的特性

### 1. 双向性

流是**双向**的，客户端和服务器都可以在同一个流上发送数据帧：

- 客户端发送 `HEADERS` 帧 + `DATA` 帧（请求）
- 服务器发送 `HEADERS` 帧 + `DATA` 帧（响应）

### 2. 有序性

在单个流内部，帧是**严格有序**的。接收端必须按照发送顺序处理帧，这样才能正确组装出完整的 HTTP 消息。

帧在流上发送的顺序很重要。接收者按照收到的顺序处理帧。特别是，报头和数据帧的顺序在语义上很重要。

### 3. 独立性

不同的流之间是**相互独立**的，可以交错发送。例如：

```
连接上的帧序列（按时间顺序）：
Stream 1: HEADERS
Stream 3: HEADERS
Stream 1: DATA
Stream 5: HEADERS
Stream 3: DATA
Stream 1: DATA (END_STREAM)
Stream 5: DATA
...
```

### 4. 虚拟性

**流（Stream）没有任何独立的物理网络实体，它是 HTTP/2 协议层凭空抽象出来的「逻辑双向通道」，底层 TCP 栈、操作系统内核完全感知不到它的存在**。

::: tip 为什么说"流"是虚拟的？

从传输的角度来看，TCP 连接上只有一个个独立的帧，每个帧携带一个流 ID 来标识它属于哪个流。**流本身并不作为一个实体存在**，它只是逻辑上的概念。

接收端收到帧后，根据流 ID 将帧"归类"到对应的流中，然后按顺序组装成完整的消息。

:::

### 5. 不可复用性

一个流（Stream）的生命周期是**一对一**严格对应的：**一个流只能用来发送一次请求，并接收一次对应的响应**。一旦这个请求-响应过程结束，该流就会立即关闭，无法在这个流上再次发送第二个请求。

## 流的完整生命周期

HTTP/2 的流是有生命周期的，会经历不同的状态：

### 流状态

```plaintext
                             +--------+
                     send PP |        | recv PP
                    ,--------|  idle  |--------.
                   /         |        |         \
                  v          +--------+          v
           +----------+          |           +----------+
           |          |          | send H /  |          |
    ,------| reserved |          | recv H    | reserved |------.
    |      | (local)  |          |           | (remote) |      |
    |      +----------+          v           +----------+      |
    |          |             +--------+             |          |
    |          |     recv ES |        | send ES     |          |
    |   send H |     ,-------|  open  |-------.     | recv H   |
    |          |    /        |        |        \    |          |
    |          v   v         +--------+         v   v          |
    |      +----------+          |           +----------+      |
    |      |   half   |          |           |   half   |      |
    |      |  closed  |          | send R /  |  closed  |      |
    |      | (remote) |          | recv R    | (local)  |      |
    |      +----------+          |           +----------+      |
    |           |                |                 |           |
    |           | send ES /      |       recv ES / |           |
    |           | send R /       v        send R / |           |
    |           | recv R     +--------+   recv R   |           |
    | send R /  `----------->|        |<-----------'  send R / |
    | recv R                 | closed |               recv R   |
    `----------------------->|        |<----------------------'
                             +--------+

       send:   端点发送此帧
       recv:   端点接收此帧

       H:  HEADERS 帧 (含隐含的延续帧)
       PP: PUSH_PROMISE 帧 (含隐含的延续帧)
       ES: END_STREAM 标记
       R:  RST_STREAM 帧
```

流总共有七种状态：

* **idle(空闲)**: 所有流都以“空闲”状态开始。

  在这种状态下，只能进行以下转换：

  * 发送或接收非推送的 `HEADERS` 帧 → **打开**
  * 发送 `PUSH_PROMISE`（服务端）, 会保留已识别的空闲流以供以后使用 → **预留(本地)**
  * 接收 `PUSH_PROMISE`（客户端），该空闲流被识别以供以后使用 → **预留(远程)**
  * **注意**:[PUSH_PROMISE](https://httpwg.org/specs/rfc7540.html#PUSH_PROMISE) 帧不在空闲流上发送，而是在 Promised Stream ID 字段中引用新保留的流。

  ::: tip

  在这种状态下，在流上接收除  [HEADERS](https://httpwg.org/specs/rfc7540.html#HEADERS) 和 [PRIORITY](https://httpwg.org/specs/rfc7540.html#PRIORITY) 之外的任何帧必须被视为 [PROTOCOL_ERROR](https://httpwg.org/specs/rfc7540.html#PROTOCOL_ERROR) 类型的连接错误

  :::

* **reserved local(预留本地)**: 处于“预留（本地）”状态的流是通过发送 [PUSH_PROMISE](https://httpwg.org/specs/rfc7540.html#PUSH_PROMISE) 帧而承诺的流。

  在这种状态下，只能进行以下转换：

  * 发送 `HEADERS` 帧→ **半关闭(本地)**
  * 任一端点都可以发送 `RST_STREAM` → **关闭**

  ::: tip

  在此状态下，端点不得发送除  [HEADERS](https://httpwg.org/specs/rfc7540.html#HEADERS), [RST_STREAM](https://httpwg.org/specs/rfc7540.html#RST_STREAM), 或 [PRIORITY](https://httpwg.org/specs/rfc7540.html#PRIORITY) 之外的任何类型的帧

  在此状态下可以接收 [PRIORITY](https://httpwg.org/specs/rfc7540.html#PRIORITY)  或 [WINDOW_UPDATE](https://httpwg.org/specs/rfc7540.html#WINDOW_UPDATE)帧。在这种状态下，在流上接收除 [RST_STREAM](https://httpwg.org/specs/rfc7540.html#RST_STREAM)、[PRIORITY](https://httpwg.org/specs/rfc7540.html#PRIORITY) 或 [WINDOW_UPDATE](https://httpwg.org/specs/rfc7540.html#WINDOW_UPDATE) 之外的任何类型的帧必须被视为 [PROTOCOL_ERROR](https://httpwg.org/specs/rfc7540.html#PROTOCOL_ERROR) 类型的连接错误

  :::

* **reserved remote(预留远程)**：处于 “reserved (remote)” 状态的流已被远程预留

  在这种状态下，只能进行以下转换：

  * 接收 `HEADERS` 帧：→ **半关闭(远程)**
  * 任一端点都可以发送 `RST_STREAM` → **关闭**

  ::: tip

  端点可以在此状态下发送 [PRIORITY](https://httpwg.org/specs/rfc7540.html#PRIORITY) 帧来重新确定保留流的优先级。在此状态下，端点不得发送除 [RST_STREAM](https://httpwg.org/specs/rfc7540.html#RST_STREAM),、[WINDOW_UPDATE](https://httpwg.org/specs/rfc7540.html#WINDOW_UPDATE) 或 [PRIORITY](https://httpwg.org/specs/rfc7540.html#PRIORITY) 之外的任何类型的帧。

  :::

* **open(打开)**: 处于“打开”状态的流可以被两方使用来发送任何类型的帧。在这种状态下，发送方应该遵守通告的流级流量控制限制

  从该状态开始，**任一端点都可以发送设置了 `END_STREAM` 标志的帧，这会导致流转换为“半关闭”状态之一**。端点发送 `END_STREAM` 标志导致流状态变为“半关闭（本地）”；接收 `END_STREAM` 标志的端点会导致流状态变为“半关闭（远程）”。

  任一端点都可以从此状态发送 [RST_STREAM](https://httpwg.org/specs/rfc7540.html#RST_STREAM) 帧，使其立即转换为“关闭”。

* **half-closed  local(半关闭 本地)**: 处于“半关闭（本地）”状态的流不能用于发送除 [WINDOW_UPDATE](https://httpwg.org/specs/rfc7540.html#WINDOW_UPDATE)、[PRIORITY](https://httpwg.org/specs/rfc7540.html#PRIORITY) 和 [RST_STREAM](https://httpwg.org/specs/rfc7540.html#RST_STREAM) 之外的帧。

  在这种状态下，可进行以下转换：

  * 接收带有 `END_STREAM` 标志的帧 → **关闭**
  * 发送或接收 `RST_STREAM` 帧→ **关闭**
  * （注意：不能再发送数据帧）

  ::: tip

  在此状态下端点可以接收任何类型的帧。

  在此状态下接收到的 [PRIORITY](https://httpwg.org/specs/rfc7540.html#PRIORITY) 帧用于重新确定取决于所识别的流的流的优先级。

  :::

* **half-closed  remote(半关闭 远程)**: 对端已发完所有数据，关闭了对端发送通道

  在这种状态下，可进行以下转换：

  * 发送带有 `END_STREAM` 标志的帧 → **关闭**
  * 发送或接收 `RST_STREAM` 帧→ **关闭**
  * （注意：不能再接收数据帧）
  
  ::: tip
  
  如果端点接收到除 [WINDOW_UPDATE](https://httpwg.org/specs/rfc7540.html#WINDOW_UPDATE)、[PRIORITY](https://httpwg.org/specs/rfc7540.html#PRIORITY) 或 [RST_STREAM](https://httpwg.org/specs/rfc7540.html#RST_STREAM) 之外的其他帧，对于处于此状态的流，它必须以  [STREAM_CLOSED](https://httpwg.org/specs/rfc7540.html#STREAM_CLOSED) 类型的流错误进行响应。
  
  在此状态下接收到的 [PRIORITY](https://httpwg.org/specs/rfc7540.html#PRIORITY) 帧用于重新确定取决于所识别的流的流的优先级。
  
  :::
  
* **closed(关闭)**: “关闭”状态是最终状态。

  * 不能发送或接收任何帧（除了 `PRIORITY` 帧），状态终结。

## 流 ID（Stream Identifier）

每个流都有一个唯一的 31 位无符号整数标识符：

### ID 分配规则

| 流 ID 范围              | 用途                                                      |
| ----------------------- | --------------------------------------------------------- |
| **0**                   | 连接控制流，用于发送连接级别的控制帧（如 SETTINGS、PING） |
| **奇数 (1, 3, 5, ...)** | 客户端发起的流                                            |
| **偶数 (2, 4, 6, ...)** | 服务器发起的流（用于服务器推送）                          |

### 特点

1. **单调递增**：新创建的流 ID 必须大于之前创建的流 ID
2. **不可复用**：流关闭后，其 ID 不能被重新使用
3. **上限限制**：最大流 ID 为 2^31 - 1，达到上限后需要新建连接

### 作用

流 ID 是多路复用的核心机制：

- 每个帧的头部都携带一个流 ID，接收端根据该 ID 将帧组装到正确的流中。
- 通过流 ID，多个并行的请求/响应可以在同一个 TCP 连接上交织传输，互不干扰。

## 流的并发控制

流的并发控制是 HTTP/2 管理在同一个 TCP 连接上同时活跃的流数量的机制，旨在防止某一端资源被耗尽，并确保公平调度。

### 核心参数：SETTINGS_MAX_CONCURRENT_STREAMS

HTTP/2 在连接建立之初，通过 **SETTINGS 帧**交换双方的配置参数。其中一个关键参数是：

- **`SETTINGS_MAX_CONCURRENT_STREAMS`**（默认值无限制，通常由实现设置）
  - 该值表示一个端点在任意时刻**允许对方发起的最大并发流数量**。
  - 例如，客户端发送 SETTINGS 帧告诉服务器：“你最多可以同时向我发起 100 个推送流”；服务器同样可以告诉客户端：“你最多可以同时向我发起 100 个请求流”。
  - 这个限制是**双向独立**的，即发送方和接收方都可以分别设置自己的限制。

::: tip 为什么需要限制？

1. **资源保护**：防止对端创建过多流消耗服务器资源
2. **公平调度**：确保每个流都能获得合理的传输机会
3. **DoS 防护**：避免恶意客户端创建大量流进行攻击

:::

## 流量控制

流量控制是 HTTP/2 中防止发送方数据淹没接收方的重要机制。它通过**窗口**限制单个流或整个连接上可以发送的数据量（DATA 帧），确保接收方有足够缓冲处理数据。

### 基本原理

- **基于窗口**：每个流和整个连接都有一个**发送窗口**（由接收方公布）。发送方只能发送窗口大小以内的数据，每发送一定数据量，窗口就相应减小。
- **动态更新**：接收方处理完数据后，通过 `WINDOW_UPDATE` 帧**增加窗口**，允许发送方继续发送更多数据。
- **独立方向**：流量控制是**单向**的。发送方对接收方进行控制，接收方通过窗口大小限制发送方的速率。

### 关键帧与参数

* 参数: `SETTINGS_INITIAL_WINDOW_SIZE`
  * 在 `SETTINGS` 帧中交换，定义**新流**的初始窗口大小。
  * 可以随时更改，但更改后会重新调整所有现有流的窗口（可能导致窗口变为负值，此时发送方必须暂停发送，直到收到足够的 `WINDOW_UPDATE`）。
* **WINDOW_UPDATE 帧**
  * 格式：`WINDOW_UPDATE` 帧包含一个 31 位的**窗口增量**（1 到 2³¹-1）。
  * 作用：
    - 流 ID = 0：增加**连接级**窗口。
    - 流 ID ≠ 0：增加对应**流级**窗口。
  * 接收方可以随时发送，但不能使窗口超过 2³¹-1（约 2GB）。

### 流量控制与并发控制的关系

| 对比项       | 流量控制                                                    | 并发控制                           |
| :----------- | :---------------------------------------------------------- | :--------------------------------- |
| **目的**     | 防止数据发送过快导致缓冲区溢出                              | 限制同时活跃的流数量，避免资源耗尽 |
| **控制对象** | 数据量（DATA 帧）                                           | 流的数量                           |
| **参数**     | 窗口大小（`WINDOW_UPDATE`、`SETTINGS_INITIAL_WINDOW_SIZE`） | `SETTINGS_MAX_CONCURRENT_STREAMS`  |
| **作用层级** | 连接级 + 流级                                               | 连接级                             |
| **动态性**   | 随时可调整窗口                                              | 可以动态修改，但已存在的流不受影响 |



## 流优先级（Stream Priority）

流优先级是 HTTP/2 为多路复用设计的**带宽分配调度机制**：客户端通过帧给流设置「依赖关系 + 权重」，告诉服务端多个并发流的传输先后顺序与带宽占比，让关键资源优先传输，避免多路复用下核心请求被非关键请求阻塞。

### 优先级的作用

- 浏览器可以告诉服务器：HTML 文档比图片更重要
- 关键 CSS/JS 应该优先于次要资源加载
- 用户可见区域的内容优先于不可见区域

::: warning 注意

HTTP/2 的优先级机制是**建议性**的，服务器可以选择是否遵守。实际实现中，很多服务器对优先级的支持并不完善。

:::

### 优先级模型: 依赖树 + 权重

HTTP/2 优先级不是简单的「高 / 中 / 低」分级，而是通过**依赖树**定先后、**权重**定带宽占比

1. **依赖树：定传输先后**

   1. 所有流的根节点是 **Stream 0（连接级）**，每个流可以指定一个父流，形成树形依赖关系。
   2. 核心规则：**父流绝对优先于子流传输**，只有父流传输完成 / 关闭，才会给子流分配带宽；同层级的流，按权重分配带宽。
   3. 特殊情况：可以设置流为「独占依赖」，独占父流的所有带宽，其他兄弟流完全不分配。

2. **权重：定同层级带宽占比**

   1. 权重取值范围：**1~256** 的整数，数值越大，优先级越高。

   2. 核心规则：同层级的兄弟流，按权重比例分配可用带宽。

      例：同层级 Stream 1 权重 256、Stream 3 权重 128，两者带宽分配比例为 2:1。

HTTP/2 使用**树形结构**来表示流的依赖关系：

```
           Root
          /    \
        A(256)  B(128)
        /   \
     C(64)  D(192)
```

- **父依赖**：流可以依赖另一个流
- **权重**：1-256 之间的整数，表示相对于同级流的资源分配比例
- **独占标志**：表示该流是父流的唯一依赖

### 核心操作与规则

1. **优先级设置时机**
   - 流创建时：在 `HEADERS` 帧里附带优先级参数，创建流的同时指定依赖和权重。
   - 运行时动态修改：通过专门的 `PRIORITY` 帧，随时调整任意流的依赖 / 权重，无需等待流关闭。
2. **关键边界规则**
   - 优先级是**客户端给服务端的建议**，不是强制命令，服务端可以遵守，也可以忽略。
   - 优先级**仅作用于 DATA 帧**：HEADERS、SETTINGS、WINDOW_UPDATE 等控制帧不受优先级约束，永远优先传输，保证协议核心逻辑不被阻塞。
   - 优先级与流 ID 大小无关：不是 Stream ID 越小，优先级越高。
   - 优先级与流量控制解耦：流量控制决定「能不能发数据」，优先级决定「先发哪个流的数据」。

## 流与帧的关系

在 HTTP/2 协议中，**流**与**帧**是构成多路复用的两个核心概念，它们的关系可以概括为：**帧是流的最小传输单元，流是承载消息的虚拟通道**。

| 概念            | 定义                                                         | 职责                                               |
| :-------------- | :----------------------------------------------------------- | :------------------------------------------------- |
| **帧 (Frame)**  | HTTP/2 通信的最小单位，固定格式的二进制结构                  | 承载协议控制信息或实际数据（如 HTTP 头部、消息体） |
| **流 (Stream)** | 连接中独立的、双向的帧序列，用于传输一个完整的 HTTP 消息（请求/响应） | 作为逻辑通道，将属于同一消息的帧有序组织并保持隔离 |

### 关联方式：Stream ID

- 每个帧的头部都包含一个 **流标识符 (Stream ID)**，长度 31 位。
- 接收端根据 Stream ID 将帧归类到对应的流中。
- 同一流上的所有帧都携带相同的 Stream ID，确保了帧与流的一对多映射（一个流包含多个帧）。

### 帧与流的交互

1. **帧是流的“零件”**

   - 一个 HTTP 请求或响应被拆分为多个帧（如一个 `HEADERS` 帧携带头部，多个 `DATA` 帧携带消息体）。
   - 这些帧按顺序发送，接收端根据帧头中的 Stream ID 将它们重新组装成完整的消息。

2. **流的生命周期由帧驱动**

   流的七种状态转换均通过特定帧的发送或接收触发：

   - `HEADERS` 帧 → 从**空闲**进入**打开**
   - 带 `END_STREAM` 标志的帧 → 进入**半关闭**
   - `RST_STREAM` 帧 → 立即进入**关闭**
   - 服务端推送使用 `PUSH_PROMISE` 帧建立**预留**流

3. **流隔离与并发**

   - 不同流的帧可以在连接上**交错传输**（如：流 1 的 DATA 帧、流 3 的 HEADERS 帧、流 1 的另一个 DATA 帧）。
   - 每个流独立维护自己的状态、流量控制窗口和优先级信息。
