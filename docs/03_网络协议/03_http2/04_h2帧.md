# 帧(Frame)

帧（Frame）是 HTTP/2 通信的**最小传输单位**，是一个固定格式的二进制结构。HTTP/2 将原本的"Header+Body"消息打散为多个二进制帧，实现高效传输。

## 为什么需要帧？

在 HTTP/1.x 中，报文是**文本形式**的，存在以下问题：

- **解析效率低**：文本格式需要逐字符解析，**边界判断复杂**
- **体积冗余**：大量重复的头部字段、换行符等占用带宽
- **无法多路复用**：必须等待前一个请求-响应完成才能处理下一个

HTTP/2 通过**二进制帧**解决这些问题：

- **解析高效**：固定格式，字段位置和长度明确，解析简单快速
- **体积紧凑**：二进制编码，无冗余字符
- **支持多路复用**：帧可以交错传输，不同请求的帧混合在一个连接中

## 帧的通用格式

所有 HTTP/2 帧都遵循统一的格式，由 **帧头(Frame Header)和帧载荷(Frame Payload)** 组成：

```plaintext
+------------------------------------------------
|                 Length (24位/3字节)           |
+------------------------------------------------
|   Type (8位/1字节)    |   Flags (8位/1字节)   |
+-------------------------------------------------------------
|R(1位)|                 Stream Identifier (31位)             |
+-+--------------------------------------------------------------
|                   Frame Payload (可变长度，0~2^24-1)         ...
+----------------------------------------------------------------
```

### 帧头字段详解

| 字段                  | 长度  | 说明                                                                |
| --------------------- | ----- | ------------------------------------------------------------------- |
| **Length**            | 24 位 | 帧载荷的长度，最大值 2^24-1（16MB-1），但实际 DATA 帧受流量控制限制 |
| **Type**              | 8 位  | 帧类型，决定载荷的格式和语义                                        |
| **Flags**             | 8 位  | 帧标志位，不同帧类型有不同含义                                      |
| **R**                 | 1 位  | 保留位，必须为 0，接收时忽略                                        |
| **Stream Identifier** | 31 位 | 流标识符，0 表示连接级帧，非 0 表示流级帧                           |

::: tip 帧头固定 9 字节

无论载荷多长，帧头始终是 **9 字节**（3+1+1+4）。这使得解析器可以快速定位帧边界。

:::

## 帧类型

HTTP/2 定义了 10 种帧类型：

| 类型值 | 帧类型        | 作用范围    | 说明                                      |
| ------ | ------------- | ----------- | ----------------------------------------- |
| 0x0    | DATA          | 流级        | 传输 HTTP 消息体数据                      |
| 0x1    | HEADERS       | 流级        | 传输 HTTP 头部，开启新流                  |
| 0x2    | PRIORITY      | 流级        | 设置流的优先级依赖和权重                  |
| 0x3    | RST_STREAM    | 流级        | 异常终止流                                |
| 0x4    | SETTINGS      | 连接级      | 协商连接参数                              |
| 0x5    | PUSH_PROMISE  | 流级        | 服务器推送承诺                            |
| 0x6    | PING          | 连接级      | 连接保活和延迟测量                        |
| 0x7    | GOAWAY        | 连接级      | 通知对端停止创建新流，优雅关闭连接        |
| 0x8    | WINDOW_UPDATE | 流级/连接级 | 流量控制窗口更新                          |
| 0x9    | CONTINUATION  | 流级        | 延续 HEADERS 或 PUSH_PROMISE 的头部块传输 |

### SETTINGS 帧

**SETTINGS 帧是 HTTP/2 连接级参数协商的唯一标准方式**，两端在连接建立初期必须互相发送 SETTINGS 帧，同步连接配置（流量控制、并发流、服务器推送等）

#### 核心属性

1. **流 ID 铁律**：必须为 `0`，纯连接级控制帧，**绝对不能用于业务流**，不属于任何 Stream ID > 0 的流。

2. **交互时机**：
   1. 在连接前言（`PRI * HTTP/2.0...`）之后，客户端和服务端都必须在任何业务流创建之前发送一个 **初始 SETTINGS 帧**，是 HTTP/2 连接建立的必经步骤。
   2. **作为对端 SETTINGS 帧的确认（ACK）**
   3. 连接运行期间动态调整（可选）： 任何一方都可以在连接存活期间的 **任意时刻** 主动发送新的 SETTINGS 帧，以动态调整连接参数。

3. **双向确认机制**：
   - 一端发送 SETTINGS 帧（不带 ACK），同步自己的配置；
   - 另一端收到后，必须**立即**发送带 `ACK` 标志的 SETTINGS 帧作为确认，不能延迟；
   - 只有双方都收到对方的 SETTINGS 帧并确认后，连接才算真正就绪，才能创建业务流。

   ![image-20260323100543765](/img/349.png)

#### Type(固定`0x04`)

固定为 `0x04`，代表 `SETTINGS` 帧。

#### Flags(标志位)

- `0x00`: 该帧包含具体的参数设置。接收方处理后，必须回复一个 ACK 帧。
- `0x01`: 确认帧，**载荷必须为空**，表示已成功应用之前接收到的 SETTINGS 参数。
- 如果收到的 SETTINGS 帧既有 ACK 标志又有非空载荷，接收方应视为 **PROTOCOL_ERROR(协议错误)**。

#### Stream Identifier(流ID)

必须为 `0`，纯连接级控制帧，**绝对不能用于业务流**，不属于任何 Stream ID > 0 的流。

#### 载荷

设置格式:

```plaintext
+-------------------------------+
|       Identifier (16)         |
+-------------------------------+-------------------------------+
|                        Value (32)                             |
+---------------------------------------------------------------+
```

- Identifier: 设置的属性ID, `2字节/16位`

- Value: 设置的值, `4字节/32位`

- **核心协商参数**:

  | 参数 ID (Identifier) | 参数名称                          | 核心作用                                                                         | 取值范围                     | 默认值                               |
  | -------------------- | --------------------------------- | -------------------------------------------------------------------------------- | ---------------------------- | ------------------------------------ |
  | **0x01**             | `SETTINGS_HEADER_TABLE_SIZE`      | HPACK 头部压缩表的最大大小，限制两端用于压缩 HTTP 头的字典大小，减少头部体积。   | 0 ~ 2^32-1                   | 4096 (4KB)                           |
  | **0x02**             | `SETTINGS_ENABLE_PUSH`            | 是否开启服务端推送（Server Push）。                                              | 0（关闭）或 1（开启）        | 1（开启）                            |
  | **0x03**             | `SETTINGS_MAX_CONCURRENT_STREAMS` | 最大并发流数量，限制对端在一条连接上最多同时开多少个业务流（流并发控制）。       | 0 ~ 2^32-1                   | 无限制（但实际实现通常设为 100/128） |
  | **0x04**             | `SETTINGS_INITIAL_WINDOW_SIZE`    | 流级初始流量控制窗口大小，所有新建流的初始窗口都用这个值，是流量控制的核心参数。 | 0 ~ 2^31-1                   | 65535 (64KB)                         |
  | **0x05**             | `SETTINGS_MAX_FRAME_SIZE`         | 单帧载荷的最大长度，限制 DATA/HEADERS 等帧的载荷大小。                           | 16384 (16KB) ~ 2^24-1 (16MB) | 16384 (16KB)                         |
  | **0x06**             | `SETTINGS_MAX_HEADER_LIST_SIZE`   | 单次请求 / 响应的头部列表最大总大小，防止超大头部攻击。                          | 0 ~ 2^32-1                   | 无限制                               |

### HEADERS 帧

HEADERS 帧是 HTTP/2 中用于传输 HTTP 头部（请求头或响应头）的核心帧类型。它既用于**开启一个新的流**，也可以在流已打开时**发送额外的头部信息**（例如 100 Continue 中间响应或 trailer 头部）。

#### 核心属性

1. **头部专属承载**：是 HTTP/2 请求 / 响应头的标准承载帧，彻底替代 HTTP/1.x 的「请求行 / 响应行 + 明文头部」格式，所有 HTTP 语义头必须通过它传输。
2. **双属性定位**：同时兼具「控制帧」+「业务承载帧」属性 —— 既控制流的生命周期、优先级调度，又承载业务请求的核心元数据。
3. **无流量控制约束**：和其他控制帧一致，完全不受 `WINDOW_UPDATE` 流量窗口限制，即使连接 / 流级窗口为 0，也可正常发送，保证协议核心逻辑不被阻塞。
4. **强压缩规则约束**：承载的头部必须经过 HPACK 算法压缩，禁止传输明文头部，是 HTTP/2 头部压缩能力的落地载体。

#### 工作原理

**发送端核心流程**

1. **头部封装与压缩**：先按规范组织 HTTP 头（强制伪头在前，常规头在后），再通过 HPACK 算法压缩成完整的 `Header Block`（头部块）。
2. **标志位与参数配置**：根据业务场景设置标志位（如 GET 请求无请求体，开启 `END_STREAM`）、优先级配置、填充规则。
3. **流 ID 分配**：为新请求分配**未使用过、符合奇偶规则、递增**的 Stream ID，初始化流的上下文。
4. **分片处理**：若压缩后的 `Header Block` 超过单帧最大长度限制（默认 16KB），拆分为多个 `Header Block Fragment`：首段放入 HEADERS 帧，后续片段用 CONTINUATION 帧续传，仅最后一个片段所在帧开启 `END_HEADERS` 标志。
5. **发送写入**：封装为完整帧，写入 TCP 连接传输。

**接收端核心流程**

1. **合法性校验**：校验帧的 Stream ID 规则、标志位合法性、长度合规性，非法帧直接返回协议错误。
2. **流状态初始化**：若为新 Stream ID，初始化流的生命周期状态与调度上下文。
3. **可选字段解析**：若开启 `PRIORITY` 标志，解析优先级字段，更新流的调度规则；若开启 `PADDED` 标志，跳过填充数据。
4. **头部收集与解码**：持续收集 `Header Block Fragment`，直到收到带 `END_HEADERS` 标志的帧，拿到完整的 `Header Block` 后，通过 HPACK 算法还原为明文 HTTP 头。
5. **流状态更新**：根据 `END_STREAM` 标志，更新流的生命周期状态（如直接进入 `half-closed` 半关闭状态）。
6. **上层交付**：把解码后的 HTTP 头交付给上层应用处理。

#### Type(固定 `0x01`)

固定为 `0x04`，代表 `HEADERS` 帧。

#### Flags(标志位)

| 标志            | 值   | 含义                                                         |
| :-------------- | :--- | :----------------------------------------------------------- |
| **END_STREAM**  | 0x1  | **表示该 HEADERS 帧是流的最后一个帧（即没有后续的 DATA 帧）。设置此标志后，发送端进入“半关闭”状态**。 |
| **END_HEADERS** | 0x2  | 表示该 HEADERS 帧包含了完整的头部块，不需要后续 CONTINUATION 帧。如果未设置，则头部块会被拆分为多个 CONTINUATION 帧。 |
| **PADDED**      | 0x8  | 表示载荷末尾填充了若干字节的填充数据（用于混淆帧长度，提高安全性）。填充长度由载荷的第一个字节指定。 |
| **PRIORITY**    | 0x20 | 表示该 HEADERS 帧的载荷中包含优先级字段（依赖和权重）。仅在流首次打开时（即流的第一个 HEADERS 帧）可以设置此标志。 |

#### Stream Identifier(流ID)

不能使用 0 值，**奇偶分配规则**：

- 客户端主动发起的请求流：必须使用**奇数 ID**（1、3、5、7...），最小可用 ID 为 1（对应你抓包的 Stream ID=1）；
- 服务端推送的流：必须使用**偶数 ID**（2、4、6、8...），且必须先通过 PUSH_PROMISE 帧锁定 ID，才能发送对应 HEADERS 帧。

#### 与 CONTINUATION 帧的关系

当 HEADERS 帧无法容纳整个压缩头部块时（受 `SETTINGS_MAX_FRAME_SIZE` 限制），必须将头部块拆分到多个 **CONTINUATION 帧**中。规则如下：

- HEADERS 帧不设置 `END_HEADERS` 标志，表示后续还有 CONTINUATION 帧。
- 紧随其后的 CONTINUATION 帧必须与 HEADERS 帧使用相同的流 ID，并依次携带剩余头部块数据。
- 最后一个 CONTINUATION 帧必须设置 `END_HEADERS` 标志，表示头部块结束。

这种拆分对 HPACK 解压是透明的，接收方会将所有分片按序组合后统一解压。

#### 载荷

HEADERS 帧的载荷是**可变结构**，各部分的存在与否由 Flags 标志位决定，且顺序绝对不可调换：

```plaintext
 +---------------+
 |Pad Length? (8)|
 +-+-------------+-----------------------------------------------+
 |E|                 Stream Dependency? (31)                     |
 +-+-------------+-----------------------------------------------+
 |  Weight? (8)  |
 +-+-------------+-----------------------------------------------+
 |                   Header Block Fragment (*)                 ...
 +---------------------------------------------------------------+
 |                           Padding (*)                       ...
 +---------------------------------------------------------------+
```

* `Pad Length? (8)`: 可选，1 字节(8 位)
  * 仅当 Flags 开启 `PADDED(0x08)` 时存在，为无符号整数，标记末尾填充数据的字节长度。
  * 核心作用：配合填充数据，混淆头部真实长度，防御基于流量分析的攻击，接收方必须忽略该字段对应的填充内容。
* `E`: 可选, 1 位
  * 仅当 Flags 开启 `PRIORITY(0x20)` 时存在
  * **取值含义**：
    - `1`（True）：**独占依赖**，该流会独占父流的所有带宽，其他同层级兄弟流完全不分配带宽；
    - `0`（False）：非独占依赖，和兄弟流按权重比例分配带宽。
* `Stream Dependency? (31)`: 可选, 31 位
  * 仅当 Flags 开启 `PRIORITY(0x20)` 时存在
  * **核心含义**：该流**依赖的父流 ID**。
* `Weight? (8)`: 可选, 固定 8 位（1 字节）
  * 仅当 Flags 开启 `PRIORITY(0x20)` 时存在。
  * **核心含义**：同层级流的**带宽分配权重编码值**。
* `Header Block Fragment (*)`: **唯一必选字段**，可变长度
  * 无论 Flags 如何都必须存在。
  * **核心含义**：经过 **HPACK 算法压缩后的 HTTP 头部二进制片段**，是 HEADERS 帧的核心业务承载。
  * **关键规则**：
    - 完整的 HTTP 头先压缩为 `Header Block`（头部块），再拆分为 1 个或多个 `Fragment`；
    - HEADERS 帧承载第一个 `Fragment`，后续片段用 CONTINUATION 帧续传；
    - 仅当 Flags 开启 `END_HEADERS(0x04)` 时，该 `Fragment` 才是完整头部的最后一段，接收方收全后才能开始 HPACK 解码。
* `Padding (*)`: 可选, 可变长度
  * 仅当 Flags 开启 `PADDED(0x08)` 时存在。
  * **长度**：可变长度，**长度 = 前面 `Pad Length` 字段的值**。
  * **内容**：无意义的 `0x00` 字节填充。
  * **核心作用**：**混淆载荷真实长度**，防御基于流量分析的攻击（比如通过头部长度推测请求类型），提升安全性。
  * **接收方处理规则**：必须完全忽略填充内容，仅根据 `Pad Length` 跳过即可。

### DATA 帧

DATA 帧是 HTTP/2 中用于传输**消息体（payload）** 的核心帧类型，承载着实际的应用数据（如 HTML、CSS、图片等）。它受流量控制机制约束，**是唯一直接受窗口限制的帧**。

#### 核心属性

- **用途**：传输 HTTP 消息体（请求或响应的实体数据）
- **流量控制**：受连接级和流级窗口的双重限制（发送前必须确保窗口充足）
- **分片**：一个完整的消息体可以被拆分为多个 DATA 帧，按序发送
- **流归属**：必须属于某个已打开的流（Stream ID ≠ 0）
- **有序性**：同一流上的 DATA 帧严格按照发送顺序传输，接收端按序组装

#### 工作原理

1. 发送方在流建立（HEADERS 帧）后，开始发送 DATA 帧。
2. 每发送一帧，会**扣减连接级 + 流级流量窗口**。
3. 接收方按 StreamID 归并数据，缓存后交给上层应用。
4. 接收方处理完数据后，通过 **WINDOW_UPDATE** 恢复窗口，允许继续发送。
5. 最后一片数据帧携带 **END_STREAM**，表示本方发送结束，流进入半关闭。

#### Type(固定 `0x00`)

固定为 `0x04`，代表 `DATA` 帧。

#### Flags(标志位)

| 标志           | 值    | 含义                                                         |
| :------------- | :---- | :----------------------------------------------------------- |
| **END_STREAM** | `0x1` | 表示该 DATA 帧是当前流的最后一个帧。设置后，发送端进入半关闭状态（不再发送数据）。 |
| **PADDED**     | `0x8` | 表示载荷中包含填充数据。设置后，载荷开头会有一个 **Pad Length** 字段（1 字节），末尾对应长度的填充字节。填充内容可任意（通常为 0），用于混淆帧长度，提升安全性。 |

#### Stream Identifier(流ID)

**必须 ≠ 0**，且必须是一个已存在的流 ID（通常为奇数，表示客户端发起的流）。

#### 载荷

结构如下: 

```text
 +---------------+
 |Pad Length? (8)|
 +---------------+-----------------------------------------------+
 |                            Data (*)                         ...
 +---------------------------------------------------------------+
 |                           Padding (*)                       ...
 +---------------------------------------------------------------+
```

| 字段           | 长度   | 出现条件          | 说明                                                         |
| :------------- | :----- | :---------------- | :----------------------------------------------------------- |
| **Pad Length** | 1 字节 | `PADDED` 标志置位 | 指示尾部填充的字节数。值范围 0~255，且不能使总载荷超过帧大小限制。 |
| **Data**       | 可变   | 始终存在          | 实际的消息体数据。如果设置了填充，其起始位置在 Pad Length 之后。 |
| **Padding**    | 可变   | `PADDED` 标志置位 | 填充字节（通常为 0x00），接收方必须忽略。                    |

### WINDOW_UPDATE 帧

WINDOW_UPDATE 帧用于**流量控制**，通知对端增加发送窗口（控制 DATA 帧），与 TCP 流量控制互补、互不干扰。

#### 核心属性

1. **双层级流 ID 规则**：
   - **`Stream ID = 0`**：连接级 WINDOW_UPDATE，作用于整条 TCP 连接，限制所有流的 DATA 帧总发送量；
   - **`Stream ID > 0`**：流级 WINDOW_UPDATE，仅作用于对应 ID 的单个流，限制该流的 DATA 帧发送量。
2. **仅作用于 DATA 帧**：所有控制帧（HEADERS、SETTINGS、RST_STREAM 等）完全不受流量控制约束，无论窗口是否为 0，都可以正常发送，保证协议核心逻辑不被阻塞。
3. **可随时发送**：`WINDOW_UPDATE` 帧可以在连接的任意时间发送，无需等待特定时机，动态调整窗口大小。

#### 工作原理

1. **初始窗口大小**:
   1. 所有流的**流级初始窗口默认值为 65535 字节（64KB）**；

   2. 连接级初始窗口默认值也为 65535 字节（64KB）。

      两端可以通过 `SETTINGS`帧的`SETTINGS_INITIAL_WINDOW_SIZE`参数，修改后续新建流的流级初始窗口大小，但连接级初始窗口只能通过 `WINDOW_UPDATE` 帧动态扩容。

2. **窗口消耗**: 发送方每发送一个 DATA 帧，就会**同步扣减对应层级的剩余窗口大小**，扣减数值 = DATA 帧的载荷字节数

3. **窗口为 0 的规则**: 当任意一层的剩余窗口为 0 时，发送方**绝对不能发送任何 DATA 帧**，但可以正常发送 HEADERS、SETTINGS、RST_STREAM 等控制帧。

4. **窗口更新（核心动作）**: 接收方处理完数据、释放缓冲区后，会发送 **WINDOW_UPDATE 帧**，告诉发送方「我的缓冲区又腾出了 X 字节，你可以继续发了：
   - 发送方收到后，对应层级的剩余窗口会加上 `Window Size Increment` 的值，恢复发送额度；
   - 只有当**连接级窗口和流级窗口都有剩余额度**时，发送方才能发送 DATA 帧。

#### Type(固定 `0x08`)

固定为 `0x04`，代表 `WINDOW_UPDATE` 帧。

#### Flags(标志位)

无有效标志，固定为 `0x00`。

#### Stream Identifier(流ID)

**双层级流 ID 规则**，根据流ID不同, 控制对象不同：

- **`Stream ID = 0`**：连接级 WINDOW_UPDATE，作用于整条 TCP 连接，限制所有流的 DATA 帧总发送量；
- **`Stream ID > 0`**：流级 WINDOW_UPDATE，仅作用于对应 ID 的单个流，限制该流的 DATA 帧发送量。

#### 载荷

有效负载格式: `WINDOW_UPDATE` 帧的有效负载是一个保留位加上一个无符号 31 位整数，指示发送方除了现有的流量控制窗口之外还可以传输的八位字节数。流量控制窗口增量的合法范围是 1 到 2<sup>31</sup> - 1 (2,147,483,647) 个八位字节。

```plaintext
+-+-------------------------------------------------------------+
|R|              Window Size Increment (31)                     |
+-+-------------------------------------------------------------+
```

### PING 帧

**PING 帧是 HTTP/2 连接级的心跳保活、往返时延（RTT）精准测量、连接可用性校验的标准控制帧**，是应用层的连接健康检测机制，与内核层的 TCP keepalive 互补，不受流量控制约束，是 HTTP/2 连接稳定性保障的核心帧之一。

#### 核心属性

1. **流 ID 铁律**：必须为 `0`，纯连接级控制帧，**绝对不能用于业务流**，不属于任何 Stream ID > 0 的流。

2. **帧类型固定**：通用帧头的 `Type` 字段固定为 `6`。

3. **无流量控制限制**：和其他控制帧一致，PING 帧完全不受流量控制窗口约束，无论连接 / 流级窗口是否为 0，都可以随时发送。

4. **与 TCP keepalive 的区别**：

   | 特性     | HTTP/2 PING 帧                           | TCP keepalive                                         |
   | -------- | ---------------------------------------- | ----------------------------------------------------- |
   | 层级     | 应用层（HTTP/2 协议栈）                  | 操作系统内核层（TCP 栈）                              |
   | 可控性   | 可自定义发送频率、即时触发、精准计时     | 内核全局配置，默认超时时间长（通常 2 小时），灵活性差 |
   | 核心能力 | 精准计算应用层 RTT、保活、连接可用性校验 | 仅底层 TCP 连接保活，无法感知应用层服务状态           |
   | 响应规则 | RFC 强制要求收到后**立即回复**           | 内核自动响应，应用层无感知                            |

#### 工作机制

PING 帧的工作逻辑是严格的「一问一答」：

1. **发起 PING 请求**：检测方（客户端 / 服务端均可主动发起）发送 **不带 ACK 标志的 PING 帧**，填充 8 字节自定义数据（通常是时间戳、随机数），同时记录发送时间。
2. **强制立即响应**：接收方收到 PING 请求帧后，**必须立即回复带 ACK 标志的 PING 响应帧**，不能延迟、不能排队、不能修改 8 字节载荷，必须原封不动返回。
3. **结果计算 / 校验**：请求方收到 ACK 响应后，对比返回的 8 字节数据与发送的是否一致，同时用「接收时间 - 发送时间」计算出精准的应用层往返时延 RTT。
4. **异常处理**：如果请求方在超时时间内未收到 ACK 响应，即可判定连接已失效，直接关闭 TCP 连接，无需等待内核超时。

#### 应用场景

1. **连接心跳保活**： 当 HTTP/2 连接长时间空闲（无业务数据传输）时，两端通过定时发送 PING 帧，检测连接是否还存活，防止中间网关、防火墙把空闲连接掐断。
   - 典型实现：浏览器、CDN 节点通常会设置 30s~60s 的空闲心跳间隔，远快于 TCP keepalive 的默认配置。

2. **精准 RTT 时延计算**： 通过 PING 帧的请求 - 响应时间差，可以精准计算出客户端与服务端之间的应用层往返时延，用于动态调整传输策略：
   - 比如服务端根据 RTT 调整流量控制窗口大小、流优先级调度；
   - 浏览器根据 RTT 选择最优的 CDN 节点、调整并发请求策略。
   - 核心优势：8 字节载荷固定，无业务数据干扰，计时精度远高于业务请求的时延统计。

3. **连接可用性快速校验**：当网络出现波动、切换（比如 Wi-Fi 切蜂窝网）时，两端可以立即发送 PING 帧，快速校验连接是否还可用，无需等待业务请求超时：

- 如果 PING 能正常收到 ACK，说明连接正常，继续复用；
  - 如果超时无响应，立即断开旧连接，新建连接，减少业务请求的失败率。

#### Type(固定 `0x06`)

固定为 `0x04`，代表 `PING` 帧。

#### Flags(标志位)

- `0x01` = ACK（确认标志）
- `0x00` 表示非确认帧

#### Stream Identifier(流ID)

必须为 `0`，纯连接级控制帧，**绝对不能用于业务流**，不属于任何 Stream ID > 0 的流。

#### 载荷

不透明的 8 字节任意数据: **ACK 响应帧必须原封不动地返回请求帧里的这 8 字节数据，不能做任何修改**。

```plaintext
+---------------------------------------------------------------+
|                                                               |
|                      Opaque Data (64)                         |
|                                                               |
+---------------------------------------------------------------+
```

### GOAWAY 帧

GOAWAY 帧是 HTTP/2 协议规定的、整条 TCP 连接优雅关闭的标准方式。用于通知对端停止创建新流，同时保证已接收的业务流能正常完成传输，避免直接断连导致的数据丢失、请求中断，是纯连接级控制帧。

#### 核心属性

1. **流 ID 铁律**：必须为 `0`，纯连接级控制帧，**绝对不能用于业务流**，不属于任何 Stream ID > 0 的流。
2. **核心作用**：单向通知对端「我即将关闭连接，不要再创建新流，我会处理完已接收的所有合法流」。
3. 与 RST_STREAM 核心区别：
   - `RST_STREAM`：强制终止**单个流**，不影响整条连接和其他流；
   - `GOAWAY`：优雅关闭**整条 TCP 连接**，不强制中断已有流，仅禁止新流创建。

#### Type(固定 `0x07`)

固定为 `0x04`，代表 `GOAWAY` 帧。

#### Flags(标志位)

无有效标志，固定为 `0x00`。

#### Stream Identifier(流ID)

必须为 `0`，纯连接级控制帧

#### 载荷

载荷结构：

```plaintext
+-+-------------------------------------------------------------+
|R|                  Last-Stream-ID (31)                        |
+-+-------------------------------------------------------------+
|                      Error Code (32)                          |
+---------------------------------------------------------------+
|                  Additional Debug Data (*)                  ...
+---------------------------------------------------------------+
```

**字段说明**：

- **R**: 保留位, 1位, 必须为 0

* **Last-Stream-ID**：最后处理的流 ID，对端不应创建更大的流 ID
* **Error Code**：**错误码**, 关闭原因（0 表示正常关闭）
* **Debug Data**：附加调试信息, 长度可变

**常见错误码**：

| 错误码 | 名称               | 说明         |
| ------ | ------------------ | ------------ |
| 0      | NO_ERROR           | 正常关闭     |
| 1      | PROTOCOL_ERROR     | 协议错误     |
| 2      | INTERNAL_ERROR     | 内部错误     |
| 3      | FLOW_CONTROL_ERROR | 流量控制错误 |
| 7      | REFUSED_STREAM     | 拒绝流       |
| 8      | CANCEL             | 取消流       |

## 帧大小限制

### 默认限制

- **最大帧大小**：16,384 字节（16KB）
- **最小帧大小**：无限制（可以是空载荷）

### 通过 SETTINGS 调整

通过 `SETTINGS_MAX_FRAME_SIZE` 参数可以调整最大帧大小：

- 最小值：16,384（2^14）
- 最大值：16,777,215（2^24 - 1）

::: warning 注意

即使设置了更大的帧大小，DATA 帧仍然受**流量控制窗口**限制，不能超过当前可用窗口大小。

:::
