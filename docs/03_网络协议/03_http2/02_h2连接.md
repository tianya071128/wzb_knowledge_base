# HTTP/2 连接

主要分为两种:

- 公网主流场景：**h2 模式**（基于 HTTPS/TLS，浏览器强制要求）
- 内网小众场景：**h2c 模式**（明文 TCP，无 TLS，浏览器不支持）

## h2(加密 HTTPS) 连接建立

HTTP2 是**二进制分帧协议**，完全基于 TCP 传输，浏览器强制要求必须运行在 **TLS 1.2 及以上版本**，通过 **TLS 的 ALPN 扩展在 TLS 握手阶段就完成 HTTP2 协议协商**，避免额外的往返开销，同时兼容 HTTP1.1。

### 阶段 1：TCP 三次握手

和 HTTP1.1 完全一致，所有 HTTP 协议的底层都是 TCP 可靠连接

### 阶段 2：TLS 握手 + ALPN 协议协商（核心！确定用 HTTP2）

**在 TLS 握手的同时，就完成了 HTTP2 协议的协商，无需额外的 HTTP 升级请求**。

核心依赖：**ALPN（Application Layer Protocol Negotiation，应用层协议协商）**，是 TLS 协议的扩展字段，用来在加密通道建立前，协商客户端和服务器都支持的应用层协议。

#### 1. 客户端发送 Client Hello

![image-20260318151758813](/img/346.png)

关键：**ALPN 扩展字段**，按优先级列出客户端支持的应用层协议

```plaintext
# ALPN 扩展 的正式名称，类型编号是 16。
Extension: application_layer_protocol_negotiation (len=14)
Type: application_layer_protocol_negotiation (16)

# 整个 ALPN 扩展的总字节数（包含子结构）。
Length: 14

# ALPN 协议列表本身的长度（12 字节），比外层总长度少 2 字节（外层 2 字节是长度字段本身）
ALPN Extension Length: 12

# 客户端按优先级顺序告诉服务器：支持这些应用层协议，选一个。
ALPN Protocol
├── ALPN string length: 2
├── ALPN Next Protocol: h2
├── ALPN string length: 8
├── ALPN Next Protocol: http/1.1
```

#### 2. 服务器选择协议

关键：**在 ALPN 扩展中返回选中的协议`h2`**，明确告知客户端：服务器支持的协议

TLS 加密通道建立后，立刻进入 HTTP2 特有的连接初始化流程，通过**二进制帧**协商连接参数，完成最终的连接确认

## h2c（明文 HTTP）连接建立

h2c（HTTP/2 Cleartext）连接的建立有两种方式：一种是利用已有的 HTTP/1.1 机制进行**升级**，另一种是客户端明确服务器支持 HTTP/2 后直接**建立**

### 方式一：从 HTTP/1.1 升级 (Upgrade)

1. **客户端发起升级请求**：客户端发送一个带有特殊头部的 HTTP/1.1 GET 请求，表明希望切换到 h2c 协议。

   ```http
   GET / HTTP/1.1
   Host: example.com
   Connection: Upgrade, HTTP2-Settings
   Upgrade: h2c
   HTTP2-Settings: <base64编码的SETTINGS帧参数>
   ```

2. **服务器同意升级**：如果服务器支持 h2c，会返回 `101 Switching Protocols` 状态码，表示同意切换协议

   ```http
   HTTP/1.1 101 Switching Protocols
   Connection: Upgrade
   Upgrade: h2c
   ```

3. **切换协议并发送连接前言**：在 `101` 响应之后，连接正式从 HTTP/1.1 切换到 HTTP/2。此时，服务器必须先发送一个 **`SETTINGS` 帧**作为它的连接前言 。客户端在收到 `101` 响应后，也必须发送它的连接前言

### 方式二：直接连接

如果客户端已经通过其他方式（例如之前的通信、配置信息等）明确知道服务器支持 h2c，那么它可以跳过繁琐的升级步骤，直接建立 HTTP/2 连接 。

1. **直接建立 TCP 连接并发送前言**：客户端在建立 TCP 连接后，立即发送连接前言：**`PRI \* HTTP/2.0\r\n\r\nSM\r\n\r\n`** 和一个 **`SETTINGS` 帧** 。它不会等待服务器响应，可以紧接着就发送真正的请求帧（如 `HEADERS` 帧）。
2. **服务器响应前言**：服务器收到客户端的连接前言后，必须回复它自己的 **`SETTINGS` 帧**作为连接前言 。
3. **确认与通信**：和升级方式一样，双方收到 `SETTINGS` 帧后需要回复确认帧，然后连接就正式建立并开始全双工通信 。
