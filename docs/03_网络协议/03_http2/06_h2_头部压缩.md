# 头部压缩(Header Compression)

头部压缩是 HTTP/2 的核心特性之一，它使用 **HPACK 算法**对 HTTP 头部进行压缩，可以减少 50%~90% 的头部数据传输量，显著提升性能，特别是在移动网络和高延迟环境下。

## 为什么需要头部压缩？

### HTTP/1.1 的问题

在 HTTP/1.1 中，头部是**纯文本格式**，存在严重的冗余问题：

#### 1. 头部体积大

一个典型的 HTTP 请求头可能包含大量字段：

```http
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/webp,*/*;q=0.8
Accept-Language: zh-CN,zh;q=0.9,en;q=0.8
Accept-Encoding: gzip, deflate, br
Connection: keep-alive
Cookie: session_id=abc123; user_token=xyz789; preference=dark_mode
```

**问题**：单个请求的头部可能达到 **几百字节甚至 1-2KB**。

#### 2. 头部高度重复

在加载一个网页时，通常会有几十甚至上百个请求，每个请求都携带大量**相同的头部字段**：

```
请求1: User-Agent: Mozilla/5.0... (200字节)
请求2: User-Agent: Mozilla/5.0... (200字节，完全相同)
请求3: User-Agent: Mozilla/5.0... (200字节，完全相同)
...
请求50: User-Agent: Mozilla/5.0... (200字节，完全相同)

总冗余: 200字节 × 50次 = 10KB 纯粹的重复数据
```

#### 3. 无法压缩头部

HTTP/1.1 只能压缩消息体（通过 `Content-Encoding: gzip`），**头部无法压缩**：

```
HTTP/1.1 报文结构：
┌─────────────────────────────────┐
│  Header (纯文本，无法压缩)       │ ← 冗余严重
├─────────────────────────────────┤
│  Body (可使用 gzip 压缩)         │ ← 已优化
└─────────────────────────────────┘
```

## HPACK 算法

HTTP/2 使用专门的 **HPACK 算法**（定义在 RFC 7541）进行头部压缩，它是一个**有状态的压缩算法**，在客户端和服务器两端维护同步的"字典"。

### 核心思想

HPACK 的核心思想是：**用索引代替重复的字符串**。

```
第一次传输：
客户端 → 服务器: User-Agent: Mozilla/5.0...
服务器记录: 索引 62 = "User-Agent: Mozilla/5.0..."

第二次传输：
客户端 → 服务器: 索引 62
服务器查表: 索引 62 = "User-Agent: Mozilla/5.0..."
```

### 三大核心机制

HPACK 由三个核心机制组成：

#### 1. 静态字典（Static Table）

预定义的**只读字典**，包含 61 个最常见的 HTTP 头部字段：

```
索引 0: (空)
索引 1: :authority = ""
索引 2: :method = "GET"
索引 3: :method = "POST"
索引 4: :path = "/"
索引 5: :path = "/index.html"
索引 6: :scheme = "http"
索引 7: :scheme = "https"
索引 8: :status = "200"
...
索引 16: accept-encoding = "gzip, deflate"
索引 24: content-type = "text/html"
...
索引 61: www-authenticate = ""
```

**优势**：

- 无需传输完整的头部字段，只需传输**索引号**
- 客户端和服务器都内置这份字典，无需协商

**示例**：

```
传输 :method GET

HTTP/1.1 -> ":method: GET" (12字节)
HTTP/2   -> 索引 2 (1字节)

压缩率: 92%
```

#### 2. 动态字典（Dynamic Table）

在连接过程中**动态构建的字典**，存储双方传输过的头部字段：

```
初始状态: 动态字典为空

第一次请求:
传输: user-agent: Mozilla/5.0...
动作: 添加到动态字典，索引 62

第二次请求:
传输: 索引 62
动作: 直接查表，无需传输完整字符串

第三次请求:
传输: x-custom-header: custom-value
动作: 添加到动态字典，索引 63
```

**特点**：

- **双向同步**：客户端和服务器各自维护动态字典，保持一致
- **FIFO 队列**：新条目从头部插入，空间不足时从尾部移除
- **大小限制**：通过 `SETTINGS_HEADER_TABLE_SIZE` 参数限制（默认 4096 字节）

#### 3. 霍夫曼编码（Huffman Encoding）

对头部字段名和值进行**霍夫曼编码**，进一步压缩：

```
原始字符串: "Mozilla/5.0"
霍夫曼编码: 11010101 10101101 ... (更少的比特位)

压缩效果: 约 15%-30% 额外压缩
```

**原理**：

- 对 HTTP 头部中常见的字符（如字母、数字）分配较短的编码
- 对不常见的字符（如特殊符号）分配较长的编码
- 基于统计概率优化编码长度

### HPACK 的完整工作流程

#### 发送端流程

```
1. 检查静态字典
   头部 ":method GET" 在静态字典中？
   → 是：发送索引 2

2. 检查动态字典
   头部 "user-agent: Mozilla/5.0..." 在动态字典中？
   → 是：发送索引 62

3. 部分匹配
   头部名在字典中，值不在？
   → 发送：索引(头部名) + 霍夫曼编码(值)
   → 例：索引 52 (user-agent) + 霍夫曼编码("Mozilla/5.0...")

4. 完全不匹配
   头部名和值都不在字典中？
   → 发送：霍夫曼编码(头部名) + 霍夫曼编码(值)
   → 添加到动态字典（可选）

5. 更新动态字典
   如果是新头部且选择缓存，添加到动态字典头部
   如果超出大小限制，从尾部移除旧条目
```

#### 接收端流程

```
1. 接收编码后的头部块（Header Block）

2. 解码每个头部条目：
   - 如果是索引：查静态/动态字典，获取完整头部
   - 如果是索引+值：查字典获取头部名，解码值
   - 如果是名+值：解码头部名和值

3. 更新动态字典：
   - 如果发送端添加了新条目，接收端同步添加

4. 交付给上层应用：
   - 组装成完整的 HTTP 头部
```

## HPACK 解码

### 步骤

分为如下步骤:

#### 一. **初始化解码上下文**

1.  加载静态字典
2.  初始化动态字典（空，索引从 62 开始，最大大小由 `SETTINGS_HEADER_TABLE_SIZE` 协商）；

#### 二. **读取首字节，识别编码类型**

1.  这是 HPACK 解码的**核心分支点**，首字节的最高几位决定了后续的解码逻辑。

2.  定义了 **6 种编码类型**，通过首字节的前缀位区分：
    | 编码类型 | 首字节前缀 | 含义 |
    | ------------------------------------- | ---------- | -------------------------------------------------- |
    | **Indexed Header Field** | `1xxxxxxx` | 完全索引头部，键值都在静态 / 动态字典里 |
    | **Literal with Incremental Indexing** | `01xxxxxx` | 字面头部，键在字典里，值是字面，解码后加入动态字典 |
    | **Literal without Indexing** | `0000xxxx` | 字面头部，不加入动态字典 |
    | **Literal Never Indexed** | `0001xxxx` | 字面头部，永远不加入动态字典（安全敏感） |
    | **Dynamic Table Size Update** | `001xxxxx` | 更新动态字典大小 |

#### 三. **根据编码类型分支解码**

1.  **Indexed Header Field（最常见，压缩率最高）**
    1. **含义**: 完全索引头部，键值都在静态 / 动态字典里
    2. **识别条件**：首字节最高位为 `1`（二进制 `1xxxxxxx`）
    3. **解码流程**：
       1. 从首字节中提取变长整数（前缀长度 N=7），得到索引值 `I`；
       2. 判断索引范围：
          - 若 `1 ≤ I ≤ 61`：查**静态字典**，直接得到完整的键值对；
          - 若 `I ≥ 62`：查**动态字典**，索引 `I-61` 对应动态字典的条目；
       3. 把解码出的键值对加入结果列表；
2.  **Literal with Incremental Indexing（次常见）**
    1. **含义**: 字面头部，键在字典里，值是字面，解码后加入动态字典
    2. **识别条件**：首字节最高两位为 `01`（二进制 `01xxxxxx`）
    3. **解码流程**：
       1. 解码键：
          - 从首字节中提取变长整数（前缀长度 N=6），得到索引值 `I`；
          - 若 `I > 0`：查静态 / 动态字典得到键；
          - 若 `I = 0`：键是字面字符串，需要单独解码（见步骤 4）；
       2. 解码值：
          - 读取下一个字节，解码字面字符串（见步骤 4），得到值；
       3. **加入结果**：把键值对加入结果列表；
       4. **更新动态字典**：把这个键值对插入到动态字典的最前端（索引 62），原有条目后移，超出大小的条目从末尾丢弃；
3.  **其他字面类型**: 流程和分支 B 类似，区别仅在于**是否更新动态字典**：

#### 四. **字面字符串解码（通用子流程）**

无论是键还是值，只要是字面字符串，都遵循这个流程：

1.  读取字符串长度的首字节 `B`；
2.  判断霍夫曼编码：
    - 若 `B` 的最高位为 `1`：使用霍夫曼编码；
    - 若 `B` 的最高位为 `0`：使用原始 ASCII 编码；
3.  解码字符串长度：
    - 把 `B` 的最高位清零，得到前缀；
    - 用变长整数规则（N=7）解码，得到字符串长度 `L`；
4.  **读取字符串内容**：读取后续 `L` 个字节；
5.  解码内容：
    - 霍夫曼编码：用 RFC 7541 霍夫曼表逐位解码，得到明文；
    - 原始编码：直接把字节转成 ASCII/UTF-8 字符；
6.  返回解码后的字符串。

### 示例

e.g: `82 41 8a a0 e4 1d 13 9d 09 b8 f3 4d 33 87 84 ...`

1. 字节 `0x82`
   1. **编码类型识别**：
      - 二进制：`10000010`
      - 最高位是 `1` → **Indexed Header Field（索引头部字段）**
   2. **索引解码**：
      - 前缀占 1 位，剩余 7 位是变长整数（N=7）；
      - 低 7 位是 `0x02`，2 < 2^7-1（127）→ 索引值 = 2；
   3. **查静态字典**：索引 2 对应 `:method: GET`；
   4. **解码结果**：`:method: GET` ✅

2. 字节 `0x41` + `0x8a` + `a0 e4 1d 13 9d 09 b8 f3 4d 33`
   1. **编码类型识别**：
      - 字节 `0x41` 二进制：`01000001`
      - 最高两位是 `01` → **Literal Header Field with Incremental Indexing（字面头部，带增量索引，会加入动态字典）**

   2. **键解码**：
      - 前缀占 2 位，剩余 6 位是变长整数（N=6）；
      - 低 6 位是 `0x01` → 索引值 = 1；
      - 查静态字典：索引 1 对应键 `:authority`；

   3. **值解码**：
      - 下一字节是 `0x8a`，二进制 `10001010`：
        - 最高位 `1` → **霍夫曼编码**；
        - 剩余 7 位 `0x0a` → 字符串长度 = 10 字节；
      - 后续 10 字节是霍夫曼编码内容：`a0 e4 1d 13 9d 09 b8 f3 4d 33`；
      - 用 RFC 7541 霍夫曼表解码 → 值为 `localhost:8443`；

   4. **解码结果**：`:authority: localhost:8443` ✅

3. 字节 `0x87`
   1. **编码类型识别**：二进制 `10000111` → 索引头部；
   2. **索引解码**：低 7 位 `0x07` → 索引值 = 7；
   3. **查静态字典**：索引 7 对应 `:scheme: https`；
   4. **解码结果**：`:scheme: https` ✅

## 动态字典管理

### 字典大小限制

通过 `SETTINGS_HEADER_TABLE_SIZE` 参数限制动态字典的最大大小：

```
SETTINGS 帧：
  SETTINGS_HEADER_TABLE_SIZE = 4096  (默认值，4KB)

含义：动态字典最多占用 4096 字节内存
```

**计算规则**：

```
每个条目的大小 = 32 + 头部名长度 + 头部值长度

示例：
user-agent: Mozilla/5.0... (假设值长度 100)
条目大小 = 32 + 10 + 100 = 142 字节
```

### 字典更新规则

#### 1. 添加新条目

```
新条目从字典头部插入：

初始状态：
[索引 62: accept-encoding: gzip]

添加 user-agent: Mozilla/5.0... 后：
[索引 62: user-agent: Mozilla/5.0...]  ← 新条目
[索引 63: accept-encoding: gzip]        ← 旧条目索引 +1
```

#### 2. 移除旧条目

```
当字典大小超出限制时，从尾部移除：

添加前：
[索引 62: 条目A (100字节)]
[索引 63: 条目B (100字节)]
[索引 64: 条目C (100字节)]
总大小: 300 字节

添加条目D (150字节)，限制 400 字节：
→ 添加条目D (总大小 450 字节，超限)
→ 移除条目C (总大小 350 字节，符合限制)

添加后：
[索引 62: 条目D (150字节)]
[索引 63: 条目A (100字节)]
[索引 64: 条目B (100字节)]
```

#### 3. 动态调整大小

```
客户端发送 SETTINGS 帧更新限制：
SETTINGS_HEADER_TABLE_SIZE = 8192

服务器收到后：
1. 如果新限制 > 当前限制：可以添加更多条目
2. 如果新限制 < 当前限制：必须移除尾部条目直到符合限制
```

### 字典索引分配

```
完整索引空间：
┌─────────────────────────────────┐
│ 索引 1-61: 静态字典             │
├─────────────────────────────────┤
│ 索引 62+: 动态字典              │
│   62: 最新添加的条目            │
│   63: 次新条目                  │
│   ...                           │
└─────────────────────────────────┘

注意：索引 0 保留，不使用
```

## 总结

HTTP/2 的头部压缩通过 **HPACK 算法**实现了高效的头部传输：

| 核心要点       | 说明                                     |
| :------------- | :--------------------------------------- |
| **静态字典**   | 预定义 61 个常见头部，无需传输完整字符串 |
| **动态字典**   | 连接过程中动态构建，存储重复头部         |
| **霍夫曼编码** | 对字符串进一步压缩，节省 15%-30%         |
| **压缩率**     | 首次请求 60%+，重复请求 90%+             |
| **有状态**     | 客户端和服务器维护同步的动态字典         |
| **安全考虑**   | 防止前缀攻击、字典同步、内存耗尽         |

头部压缩是 HTTP/2 性能提升的重要组成部分，配合多路复用、二进制帧等特性，使得 HTTP/2 成为现代 Web 的标准协议。
