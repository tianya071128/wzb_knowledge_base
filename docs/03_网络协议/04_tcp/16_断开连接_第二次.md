# 第二次挥手

![img](/img/372.png)

## 基础定位

**发起方**：被动关闭方

**报文类型**：纯 ACK 确认报文（仅 ACK 标志置 1，无 FIN）

**核心作用**：确认收到了对方的关闭请求（FIN 报文），告知主动方「我已经收到你停止发送的通知了」

**结果**：**主动方 → 被动方的单向发送通道正式关闭**，进入半关闭状态；被动方仍可以继续向主动方发送剩余业务数据。

## 触发时机

**被动方收到了主动方发来的、合法且按序的 FIN 报文（第一次挥手）**。

**合法性校验**：内核校验报文的序列号是否在接收窗口内、是否是**按序到达**的 FIN。

- 按序合法 FIN：直接进入下一步，即时触发 ACK 回复。
- 乱序 FIN：先暂存报文，等前面缺失的数据到达补齐后，再处理 FIN、回复 ACK。

## 特殊情况：捎带确认的第二次挥手

虽然 FIN 默认立刻回纯空 ACK，但如果**收到 FIN 的同时，被动方刚好有业务数据要发给主动方**，内核会把「第二次挥手的 ACK 确认」和「业务数据」合并在同一个报文里发送，也就是**捎带确认**。

这种场景下的第二次挥手有两个特点：

1. 报文 `TCP Segment Len ≠ 0`，携带了业务数据；
2. ACK 确认的作用不变，仍然是对对方 FIN 的确认，属于第二次挥手范畴。

## 报文

![image-20260616162341443](/img/374.png)

### 源 / 目的端口

```plaintext
Source Port: 60366
Destination Port: 3000
```

- `Source Port: 60366`：**被动关闭方端口**，收到第一次挥手的 FIN 后，由内核自动回复 ACK 确认。
- `Destination Port: 3000`：主动关闭方端口，接收本次确认报文。

### 序列号（Sequence Number）

```plaintext
Sequence Number: 1135    (relative sequence number)
Sequence Number (raw): 830817877
[Next Sequence Number: 1135    (relative sequence number)]
```

- `Sequence Number: 1135(relative sequence number)`：被动方自身的发送序列号基线，承接上一次发送数据的末尾序号。
- `Sequence Number (raw): 830817877`：原始真实绝对序列号，Wireshark 默认展示相对序号简化阅读。
- `[Next Sequence Number: 1135    (relative sequence number)]`：下一个待发送序号和当前序号完全相同。
  **核心规则：**纯 ACK 报文（无数据、无 FIN/SYN）不消耗序列号；后续被动方发送业务数据、或者发送第三次挥手的 FIN 时，序号仍然从 1135 开始计算。

### 确认号（Acknowledgment Number）

```plaintext
Acknowledgment Number: 181（相对 ACK）
Acknowledgment number (raw): 96981202
```

- `Acknowledgment Number: 181（相对 ACK）`: 第一次挥手 Seq=180 + 1 = 181。TCP 协议规定：**FIN 标志位会单独占用 1 个序列号**，和 1 字节业务数据等效。

### 头部长度（Header Length）

```plaintext
0101 .... = Header Length: 20 bytes (5)
```

- `Header Length: 20 bytes (5)`：TCP 头部总长度为 20 字节
- 结构：`20 字节固定头部`。无任何 TCP 扩展选项。

### 标志位（Flags: 0x002 (SYN)）

```plaintext
Flags: 0x010 (ACK)
    000. .... .... = Reserved: Not set
    ...0 .... .... = Accurate ECN: Not set
    .... 0... .... = Congestion Window Reduced: Not set
    .... .0.. .... = ECN-Echo: Not set
    .... ..0. .... = Urgent: Not set
    .... ...1 .... = Acknowledgment: Set
    .... .... 0... = Push: Not set
    .... .... .0.. = Reset: Not set
    .... .... ..0. = Syn: Not set
    .... .... ...0 = Fin: Not set
    [TCP Flags: ·······A····]
```

- **ACK = Set（1）**

  确认标志置 1，对应上面的 ACK=1135，确认对端历史数据。

### 窗口大小（Window）

```plaintext
Window: 255
[Calculated window size: 65280]
[Window size scaling factor: 256]
```

- `Window: 251`: TCP 头部 16 位原始窗口数值
- `Calculated window size: 65280`：实际接收缓冲区空闲大小 = 255 * 256 = 65280 字节。
- `Window size scaling factor: 256`：三次握手协商的窗口缩放因子

### 校验和（Checksum）

```plaintext
Checksum: 0x68d9 [unverified]
[Checksum Status: Unverified]
```

- TCP 校验和，用来校验传输过程中比特差错；
- `unverified` 是因为网卡开启了 **TCP 校验和卸载（Checksum Offload）**，抓包时看到的校验和是硬件计算前的占位值，不是真正的错误。

### 紧急指针（Urgent Pointer）

```plaintext
Urgent Pointer: 0
```

- 因为 `URG=0`，紧急指针字段无效，值为 0。

### TCP 选项

无选项

## 状态变更

### 被动关闭方

发送完本 ACK 报文后，连接状态从 `ESTABLISHED` 切换为 **`CLOSE_WAIT`**

### 主动关闭方

收到本 ACK 报文后，连接状态从 `FIN_WAIT_1` 切换为 **`FIN_WAIT_2`**

* 本方发送通道正式关闭，不再发送新数据；
* 接收通道保持正常，继续等待被动方发送剩余数据、以及第三次挥手的 FIN 报文。
