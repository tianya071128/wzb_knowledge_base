# 为什么需要三次握手?

**三次握手是在不可靠的网络信道上，建立可靠双向连接的理论最小值**

## TCP 建立连接到底在做什么？

**TCP 建立连接的核心目的，是同步双方的初始序列号（ISN）。**

TCP 是面向字节流的协议，它给传输的**每一个字节**都分配一个唯一的序列号。所有的确认应答、超时重传、有序性保证、重复数据丢弃，全部建立在序列号之上。

如果双方没有同步彼此的初始序列号，TCP 的所有可靠机制都将完全失效。

## 为什么必须是三次？

主要有三个方面的原因:

1. **核心原因：解决 "旧的重复连接启动造成混乱" 问题**

   这是 RFC 793 原文中明确指出的、三次握手存在的最根本理由。

2. 同步双方的初始序列号

### 防止旧的重复连接启动造成混乱

#### 两次握手的致命缺陷

当旧的连接(SYN 报文)因为丢失/延迟而发起了重传时:

```plaintext
时间线 →
客户端                          服务端
  |                              |
  |  SYN(seq=1000)               |  客户端发起连接，报文在网络中滞留
  |----------------------------->|
  |  (报文丢失/延迟)              |
  |                              |
  |  超时重发 SYN(seq=2000)      |  客户端重发新的SYN
  |----------------------------->|
  |                              |
  |  SYN+ACK(seq=3000, ack=2001) |  服务端确认，建立连接
  |<-----------------------------|
  |                              |
ESTABLISHED                    ESTABLISHED
  |                              |
  |  数据传输完成                |
  |<---------------------------->|
  |                              |
  |  正常关闭连接                |
  |<---------------------------->|
  |                              |
CLOSED                         CLOSED
  |                              |
  |                              |  ⚠️  很久之后，旧的SYN(seq=1000)终于到达
  |----------------------------->|
  |                              |
  |  SYN+ACK(seq=4000, ack=1001) |  ❌  服务端误以为是新连接，再次进入ESTABLISHED
  |<-----------------------------|
  |                              |
  |  (客户端已关闭，无任何响应)   |  服务端一直维持这个无效连接
  |                              |  浪费CPU、内存、文件描述符资源
  |                              |
  |                              |  ⏳  直到超时才会释放（默认30秒以上）
```

#### 问题总结

两次握手的致命缺陷是：**服务端无法区分这个 SYN 是新的连接请求，还是网络中滞留的旧连接请求**。

只要有一个旧的 SYN 报文到达服务端，服务端就会无条件建立一个无效连接，并且一直维持到超时。在高并发场景下，这会迅速耗尽服务端的所有资源，导致正常连接无法建立。

#### 三次握手

在两次握手的情况下，**服务端没有中间状态给客户端来阻止历史连接，导致服务端可能建立一个历史连接，造成资源浪费**

在三次握手的情况下, **服务端会先进入 `SYN_RCVD` 状态, 等待客户端的回复**

总结而言: 两次握手时 **"服务端无条件信任所有 SYN 报文"**，而三次握手通过"**第三次 ACK 确认**"，让服务端只有在收到客户端的明确确认后，才会建立完整连接。

```plaintext
时间线 →
客户端                          服务端
  |                              |
  |  SYN(seq=1000)               |  客户端发起连接，报文在网络中滞留
  |----------------------------->|
  |  (报文丢失/延迟)              |
  |                              |
  |  超时重发 SYN(seq=2000)      |  客户端重发新的SYN
  |----------------------------->|
  |                              |
  |  SYN+ACK(seq=3000, ack=2001) |  服务端确认，建立连接
  |<-----------------------------|
  |                              |
  |  ACK(seq=2001, ack=3001)     |
  |----------------------------->|
  |                              |
ESTABLISHED                    ESTABLISHED
  |                              |
  |  数据传输完成                |
  |<---------------------------->|
  |                              |
  |  正常关闭连接                |
  |<---------------------------->|
  |                              |
CLOSED                         CLOSED
  |                              |
  |                              |  ⚠️  很久之后，旧的SYN(seq=1000)终于到达
  |----------------------------->|
  |                              |
  |  SYN+ACK(seq=4000, ack=1001) |  ✅  服务端进入SYN_RCVD，不建立完整连接
  |<-----------------------------|
  |                              |
  |                              |  只占用半连接队列资源
  |                              |
  |  RST(seq=1001)               |  ✅  客户端发现不是自己发起的连接，回复RST
  |----------------------------->|
  |                              |
  |                              |  ✅  服务端收到RST，立即释放半连接资源
  |                              |  不会进入ESTABLISHED
  |                              |  不会浪费完整连接资源
  |                              |
CLOSED                         LISTEN
```

### 同步双方初始序列号

**序列号是 TCP 可靠连接的基石, 所以必须要保证通信双方的序列号同步**。

两次握手只保证了一方的初始序列号能被对方成功接收，没办法保证双方的初始序列号都能被确认接收。

而三次握手才能确保双方的初始序列号能被可靠的同步

### 为什么不需要四次？

理论上可以拆成四次：客户端 SYN → 服务端 ACK → 服务端 SYN → 客户端 ACK。

但服务端的 `ACK`（确认客户端的 SYN）和 `SYN`（同步自己的序列号）可以**合并在同一个报文里发送**，因此三次就足够了，无需额外的一次交互。
