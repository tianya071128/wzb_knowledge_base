# 重传机制

TCP 重传是实现可靠传输的核心机制，它的本质是：**发送方发送数据后，TCP 针对数据包丢失的情况，会用重传机制解决**

常见的重传机制：

- 超时重传
- 快速重传
- SACK 选择性确认

## 超时重传

* 发送方每发送一个数据段，都会为它启动一个**超时重传计时器（RTO）**
* 如果在 RTO 时间内收到了该数据段的 ACK，就停止计时器
* 如果 RTO 超时仍未收到 ACK，就判定该数据段丢失，立即重传

### RTO（重传超时时间）的计算

精确的测量超时时间 `RTO` 的值是非常重要的，这可让重传机制更高效。

- 当超时时间 **RTO 较大**时，重发就慢，丢了老半天才重发，没有效率，性能差；
- 当超时时间 **RTO 较小**时，会导致可能并没有丢就重发，于是重发的就快，会增加网络拥塞，导致更多的超时，更多的超时导致更多的重发

TCP 采用**自适应超时算法**，根据网络往返时间（RTT）动态调整 RTO，不在此赘述。

### 重传的退避机制（指数退避）

TCP 每次重传超时后，会将 RTO 时间翻倍，避免加重网络拥塞：

- 第一次超时：RTO=1 秒
- 第二次超时：RTO=2 秒
- 第三次超时：RTO=4 秒
- 第四次超时：RTO=8 秒
- 以此类推，直到达到最大重传次数（Linux 默认 15 次，约 9 分钟），判定连接断开

## 快速重传

普通**超时重传**缺点：丢包后必须等**RTO 超时**才能重传，等待时间长，网速变慢、延迟高。

于是设计出**快速重传**：**不等超时，靠重复 ACK 直接重传**。

### 触发条件

1. 收到 **3 个相同的重复 ACK**
2. 无需等待超时计时器到期，**立刻重传丢失报文**
3. 只重传**缺失的那一段**，不重传后面已收到的数据

::: tip 为什么是 3 个重复 ACK

- 1 个、2 个重复 ACK：可能只是**网络乱序**，不是丢包
- 连续 3 个：基本判定**确定丢包**，误判概率极低

:::

### 案例

设：发送方依次发送 4 个报文

`SEQ=100`、`SEQ=200`、`SEQ=300`、`SEQ=400`

1. **SEQ=100 报文丢失**

2. 后面 `200、300、400` 顺利到达接收方

3. 接收方发现乱序，缺 100

   按照 TCP 规则：只回复期望收到的序号 ACK=100

   - 收到 200 → 回复 ACK=100（第 1 个重复 ACK）
   - 收到 300 → 回复 ACK=100（第 2 个重复 ACK）
   - 收到 400 → 回复 ACK=100（**第 3 个重复 ACK**）

4. 发送方集齐 **3 个重复 ACK**

5. **立即重传 SEQ=100**，不等超时

::: tip 注意

**确认号只回复当前已确定收到的连续字节 + 1**，如果后面的包先到的话，也只回复已确定的ACK

:::

## SACK 选择性确认

普通 TCP 是**累积确认**，只确认到连续收到的最远位置；

SACK 可以**单独标出哪些分片已经收到、哪些丢失**，发送方**只重传丢失的部分**，不重传已经收到的数据。

### ACK 确认机制的问题

接收方只能回复已经连续收到的最远位置，一旦触发重传 / 超时，可能会**把后续的数据包全部重传**，大量重复流量，极度浪费带宽。

### 工作原理

1. 三次握手阶段双方协商[开启 SACK 选项](/tcp/header_option.html#sack-允许-sack-permitted)
2. 乱序收到数据时，**基础 ACK + SACK 块**一起发
3. 基础 ACK：依旧是**最小缺失序号**
4. SACK 选项：带上**已经收到的不连续区间**

::: tip 格式

SACK = [左边界 , 右边界]

**标准最多携带 4 个 SACK 块**

:::

### 工作流程

当触发超时重传或快速重传机制时, 通过 `SACK` 信息发现只有部分数据丢失，则重发时，就只选择了这个 TCP 段进行重复。
