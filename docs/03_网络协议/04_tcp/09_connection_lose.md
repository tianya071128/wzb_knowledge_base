# 握手中丢失包

## 第一次握手丢失包

第一次握手 SYN 丢失，客户端不断超时重传 SYN，多次重试失败后连接建立失败，服务端无任何反应。

重传机制：

1. 超时时间：不同版本的操作系统可能超时时间不同，有的 1 秒的，也有 3 秒的

2. 重传次数： 在 Linux 里，客户端的 SYN 报文最大重传次数由 `tcp_syn_retries` 内核参数控制

3. 采用**指数退避**重试：间隔时间是上一次的 2 倍。

   例如: 1s → 2s → 4s → 8s → 16s

4. 超过重传次数后, 连接建立失败, 客户端退回 CLOSED 状态

::: tip 注意

**重传 SYN 报文，序列号不变**

序列号是**本次连接唯一初始编号**，连接没失败就不换

:::

## 第二次握手丢失包

在第二次握手中, 服务端接收到第一次握手的 SYN, 状态发生变更, 此时双方状态:

- **客户端**：停留在 **SYN_SENT**，等待 SYN+ACK
- **服务端**：进入 **SYN_RCVD**，等待第三次 ACK

**两端各自行为**:

* **客户端**: 收不到应答, 不确定第一次握手是否丢包 → 超时重传**第一次 SYN 报文**
* **服务端**: 收不到客户端第三次 ACK, 也会触发超时重传机制

## 第三次握手丢失包

此时双方状态:

* **客户端**: 发送完 ACK 直接进入：**ESTABLISHED**。认为连接已经成功，可以**直接发数据**
* **服务端**：没收到 ACK，停留在：**SYN_RCVD**。等待客户端确认。

**两端各自行为**：

* **客户端**: 进入 `ESTABLISHED`。ACK 包不会重传
* **服务端**: 收不到客户端第三次 ACK, 触发超时重传机制

**自动修复机制**:

客户端先发**数据报文**发给服务端：

1. 服务端在 SYN_RCVD 状态收到**客户端数据**
2. 自动默认收到了第三次 ACK
3. 直接切换为 **ESTABLISHED**
4. 连接恢复正常，通信畅通

::: tip 注意

**ACK 报文是不会有重传的，当 ACK 丢失了，就由对方重传对应的报文**。

:::
