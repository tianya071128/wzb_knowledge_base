## 源端口 - 16 位

- 作用: **标识发送方应用进程**
- 取值范围：0 ~ 65535
- 一般客户端使用 1024 ~ 65535 的临时端口，服务端使用 0 ~ 1023 的知名端口（如 80、443、22）

## 目的端口 - 16 位

- 作用：**标识接收方应用进程**
- 取值范围：0 ~ 65535
- 核心：**IP 找主机，端口找进程**

## 序列号 - 32 位

**序列号是 TCP 可靠传输的灵魂**，所有的有序性、去重、确认应答、超时重传机制，全部建立在序列号之上。

### 核心定义

TCP 序列号是一个 **32 位无符号整数**，取值范围 `0 ~ 4294967295`。

**最本质的规则：TCP 给传输的**每一个**字节数据**，都分配一个唯一的序列号。

::: tip 注意

**每个字节有一个序列号**，报文的序列号只是该报文第一个数据字节的编号。

例子:

如果一个 TCP 报文的 `SEQ=1000`，数据长度是 `200` 字节，那么：

- 这个报文包含的字节编号是 `1000 ~ 1199`
- 下一个报文的第一个字节编号应该是 `1200`

:::

### 核心作用

1. **保证数据按序到达**: IP 网络是不可靠的，数据包可能乱序到达。接收方通过序列号，把乱序的字节重新排序，组装成完整的应用层数据。

2. **实现数据去重**: 网络可能会重复发送同一个数据包（比如超时重传）。接收方通过序列号判断，如果某个字节已经收到过，就直接丢弃重复的部分。

3. **作为确认应答的基础**: 确认号（ACK）的含义是：**我已经收到了前 N 个字节，期望收到下一个字节的编号是 N**。

   ```plaintext
   ACK号 = 对方的序列号 + 对方的数据长度
   ```

4. **作为超时重传的基础**: 发送方维护一个重传计时器，如果在规定时间内没有收到对应字节的 ACK，就会重传该字节开始的数据。

### 初始序列号

TCP 连接建立时，双方会各自生成一个**随机的初始序列号（Initial Sequence Number, ISN）**，而不是从 0 开始。

1. **安全原因**: 防止序列号预测攻击。如果 ISN 是固定的或可预测的，攻击者可以伪造 TCP 报文，注入恶意数据。
2. **避免旧连接干扰**: 防止网络中残留的旧连接数据包，被新连接误认为是自己的数据。

**关键规则**

* **SYN 报文和 FIN 报文，即使没有数据，也要消耗一个序列号**: 所以上面的 ACK 号是 `ISN_C+1` 和 `ISN_S+1`，而不是 `ISN_C` 和 `ISN_S`
* **纯 ACK 报文（没有数据），不消耗序列号**
* SYN 和 FIN 本身是控制信息，需要可靠传输（即需要被确认）。消耗一个序列号使它们能被 ACK 单独确认，否则无法区分是否丢失。
* 纯 ACK 只用于确认对方数据，无需对方再确认，因此不消耗序列号，节省序号空间。

## 确认号 - 32 位

**确认号是 TCP 可靠传输的另一半灵魂**，它和序列号成对出现，共同构成了 TCP 确认应答机制的核心。

### 核心定义

TCP 确认号是一个 **32 位无符号整数**，和序列号一一对应。

**最本质的规则：确认号表示「我已经成功收到了前 N-1 个字节，下一个期望收到的字节编号是 N」。**

```plaintext
ACK号 = 对方上一个报文的序列号 + 对方上一个报文的数据长度
```
