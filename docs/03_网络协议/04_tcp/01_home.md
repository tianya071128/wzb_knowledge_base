# TCP

## 概述

TCP（Transmission Control Protocol，传输控制协议）是一种**面向连接的、可靠的、基于字节流的传输层通信协议**。它是互联网协议族（TCP/IP）的核心协议之一，位于传输层，为应用层提供可靠的数据传输服务。

## 主要特点

### 1. 面向连接

- 通信双方必须先通过**三次握手**建立一个逻辑连接，才能开始传输数据；通信结束后必须通过**四次挥手**断开连接
- 通信前必须建立连接（三次握手）
- 通信结束后需要释放连接（四次挥手）

### 2. 可靠传输

- 保证数据**不丢失、不重复、按序到达**，这是 TCP 最核心的价值。
- 使用确认和重传机制保证**数据可靠到达**
- 通过序号保证数据的**有序性**
- 通过校验和检测数据的**完整性**

### 3. 面向字节流

- 把应用层传来的数据看成**一串连续的、无边界的字节流**，不保留应用层的报文边界。
- **发送方和接收方需要自行处理消息边界**

### 4. 全双工通信

- 连接的双方可以在任何时候发送和接收数据
- 每个方向的传输独立管理

### 5. 端到端通信（End-to-End）

- TCP 只在**通信双方的主机**上实现，中间的路由器、交换机只负责转发 IP 数据包，不参与任何 TCP 逻辑
- 所有的连接建立、确认应答、重传、流量控制、拥塞控制，都只在发送端和接收端处理；
- 通过**IP 地址 + 端口号**唯一标识一个 TCP 连接，实现进程到进程的通信。

## TCP 四元组: 确定唯一的 TCP 连接

**TCP 四元组是唯一标识一个 TCP 连接的核心机制**

![TCP 四元组](/img/357.png)

| 字段         | 作用                       |
| ------------ | -------------------------- |
| **源 IP**    | 标识发送方主机             |
| **源端口**   | 标识发送方主机上的应用进程 |
| **目的 IP**  | 标识接收方主机             |
| **目的端口** | 标识接收方主机上的应用进程 |

源地址和目的地址的字段（32 位）是在 IP 头部中，作用是通过 IP 协议发送报文给对方主机。

源端口和目的端口的字段（16 位）是在 TCP 头部中，作用是告诉 TCP 协议应该把报文发给哪个进程。

### 为什么需要四元组?

TCP 是**端到端、全双工**的协议，一台主机可以同时和多个主机建立多条 TCP 连接，同一个主机上的多个进程也可以同时和同一个目标建立多条连接。

**只有通过四元组，操作系统才能区分不同的 TCP 连接，把收到的数据准确交给对应的应用程序。**

## TCP 与 UDP 对比

| 对比维度         | TCP                                                            | UDP                                                  |
| ---------------- | -------------------------------------------------------------- | ---------------------------------------------------- |
| **全称**         | Transmission Control Protocol（传输控制协议）                  | User Datagram Protocol（用户数据报协议）             |
| **核心设计目标** | 在不可靠的 IP 网络上提供**可靠、有序、无差错**的端到端传输     | 提供**最低开销、最低延迟**的端到端传输               |
| **连接特性**     | **面向连接**：必须先三次握手建立连接，通信结束四次挥手断开连接 | **无连接**：不需要建立连接，直接发数据包，发完就走   |
| **可靠性**       | **绝对可靠**：保证数据不丢失、不重复、按序到达                 | **不可靠**：不保证数据到达，不保证有序，不保证不重复 |
| **有序性**       | 严格保证按序到达，乱序会自动重组                               | 不保证有序，数据包可能乱序到达                       |
| **流量控制**     | 有（滑动窗口）：防止发送方发太快把接收方撑爆                   | 无：发送方想发多少发多少，不管接收方能不能处理       |
| **拥塞控制**     | 有（慢启动、拥塞避免、快速重传 / 恢复）：防止网络拥塞          | 无：不管网络堵不堵，都按固定速度发                   |
| **头部开销**     | **至少 20 字节**（可选字段最多 60 字节）                       | **固定 8 字节**，开销极小                            |
| **传输模式**     | 字节流：无报文边界，会粘包 / 拆包                              | 数据报：有明确报文边界，不会粘包                     |
| **错误处理**     | 校验和错误会丢弃并重传                                         | 校验和错误直接丢弃，不重传                           |
| **全双工**       | 支持：同一连接双向同时收发                                     | 支持：同一端口双向同时收发                           |
| **队头阻塞**     | 有：TCP 层队头阻塞，单个丢包阻塞整条连接                       | 无：数据包独立，一个丢包不影响其他                   |

## 常见应用场景

### 适用场景

- 需要可靠传输的应用
  - Web 浏览（HTTP/HTTPS）
  - 电子邮件（SMTP、POP3、IMAP）
  - 文件传输（FTP）
  - 远程登录（SSH、Telnet）
- 对数据完整性要求高的场景
- 需要顺序传输的场景

### 不适用场景

- 实时音视频传输（延迟敏感）
- 在线游戏（延迟敏感）
- DNS 查询（简单请求，开销大）
- 广播/多播应用
