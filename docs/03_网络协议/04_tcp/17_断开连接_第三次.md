# 第三次挥手

![img](/img/372.png)

## 基础定位

**发起方**：被动关闭方

**报文类型**：FIN+ACK 确认报文

**核心作用**：被动方告知主动方「我这边的业务数据也全部发送完毕了，现在关闭本方的发送通道」

**结果**：TCP 双向的发送通道均宣告关闭，进入最后一步确认阶段，连接即将彻底释放。

## 触发时机

**第三次挥手由应用层主动调用 close() 触发，而第二次挥手由内核自动完成，这是两者最本质的触发机制区别。这也决定了为什么 CLOSE_WAIT 状态会因应用层迟迟不关连接而堆积**

完整触发时序：

1. 被动方内核收到第一次挥手的 FIN → 自动回复第二次挥手 ACK → 进入 CLOSE_WAIT，同时通知应用层「对端已关闭发送」。
2. 被动方应用层收到通知后，继续处理 / 发送剩余的业务数据（比如返回响应、传输文件）。
3. **当应用层把所有业务数据都发送完毕，主动调用`close()`关闭套接字时**，内核才会构造 FIN 报文，发送第三次挥手。

## 第三次挥手的“合并”现象：三次挥手

如果被动方在进入 `CLOSE_WAIT` 后，**恰好没有数据要发送**，那么它可以把第二次挥手（ACK）和第三次挥手（FIN）合并成一个报文

## 报文

![image-20260616164201186](/img/375.png)

### 源 / 目的端口

```plaintext
Source Port: 60366
Destination Port: 3000
```

- `Source Port: 60366`：**被动关闭方端口**，当应用层调用`close()`关闭套接字时，由内核构造并发送本次 FIN 报文。
- `Destination Port: 3000`：主动关闭方端口，接收本次关闭通知报文

### 序列号（Sequence Number）

```plaintext
Sequence Number: 1135（相对序列号）
Sequence Number (raw): 830817877
[Next Sequence Number: 1136]
```

- `Sequence Number: 1135（相对序列号）`：和第二次挥手的 Seq 数值完全相同。**因为第二次挥手是**纯 ACK 报文**，不消耗任何序列号；被动方发送完第二次挥手后，发送序号基线没有变化，仍然停留在 1135**

- `Sequence Number (raw): 830817877`：原始真实绝对序列号，Wireshark 默认展示相对序号简化阅读。

- `[Next Sequence Number: 1136]`：**FIN 标志位会单独占用 1 个序列号**，等效于发送了 1 字节的控制数据。

  因此下一个待发送序号 = 当前 Seq + 1 = 1135 + 1 = 1136。

### 确认号（Acknowledgment Number）

```plaintext
Acknowledgment Number: 181（相对 ACK）
Acknowledgment number (raw): 96981202
```

- `Acknowledgment Number: 181（相对 ACK）`: 确认号与第二次挥手完全一致。

  原因：第一次挥手后，主动方已经关闭了自身发送通道，没有再发送任何新数据；**因此被动方不需要更新确认号，继续沿用 Ack=181，捎带确认对方所有已接收数据**。

### 头部长度（Header Length）

```plaintext
0101 .... = Header Length: 20 bytes (5)
```

- `Header Length: 20 bytes (5)`：TCP 头部总长度为 20 字节
- 结构：`20 字节固定头部`。无任何 TCP 扩展选项。

### 标志位（Flags: 0x002 (SYN)）

```plaintext
Flags: 0x011 (FIN, ACK)
    000. .... .... = Reserved: Not set
    ...0 .... .... = Accurate ECN: Not set
    .... 0... .... = Congestion Window Reduced: Not set
    .... .0.. .... = ECN-Echo: Not set
    .... ..0. .... = Urgent: Not set
    .... ...1 .... = Acknowledgment: Set
    .... .... 0... = Push: Not set
    .... .... .0.. = Reset: Not set
    .... .... ..0. = Syn: Not set
    .... .... ...1 = Fin: Set
    [TCP Flags: ·······A···F]
```

- **FIN = Set（1）**

  第三次挥手的核心标志，语义：**本方发送通道正式关闭，不再产生任何新的业务数据**。
  
  至此，TCP 双向的发送通道全部宣告关闭，连接进入收尾确认阶段。
  
- **ACK = Set（1）**

  捎带确认标志，标识确认号字段有效；将确认逻辑和 FIN 关闭通知合并在同一条报文中，节省网络包开销，无需单独发送纯 ACK 报文。

### 窗口大小（Window）

```plaintext
Window: 255
[Calculated window size: 65280]
[Window size scaling factor: 256]
```

- `Window: 255`: TCP 头部 16 位原始窗口数值
- `Calculated window size: 65280`：实际接收缓冲区空闲大小 = 255 * 256 = 65280 字节。
- `Window size scaling factor: 256`：三次握手协商的窗口缩放因子

### 校验和（Checksum）

```plaintext
Checksum: 0x68d8 [unverified]
[Checksum Status: Unverified]
```

- TCP 校验和，用来校验传输过程中比特差错；
- `unverified` 是因为网卡开启了 **TCP 校验和卸载（Checksum Offload）**，抓包时看到的校验和是硬件计算前的占位值，不是真正的错误。

### 紧急指针（Urgent Pointer）

```plaintext
Urgent Pointer: 0
```

- 因为 `URG=0`，紧急指针字段无效，值为 0。

### TCP 选项

无选项

## 状态变更

### 被动关闭方

发送完本次 FIN 报文后，连接状态从 **`CLOSE_WAIT`** 切换为 **`LAST_ACK`**

- 含义：最后确认状态，等待主动方回复第四次挥手的 ACK 确认报文。
- 该状态下被动方不再发送任何数据；如果超时未收到 ACK，会触发重传机制重发 FIN 报文。

### 主动关闭方

收到本次 FIN 报文后，连接状态从 **`FIN_WAIT_2`** 切换为 **`TIME_WAIT`**

* 收到 FIN 后，内核会立刻回复第四次挥手的 ACK 确认报文；
* 进入 TIME_WAIT 后，会等待 2MSL 时长，确保最后一个 ACK 可靠到达对端，同时清空链路中残留的旧报文。
