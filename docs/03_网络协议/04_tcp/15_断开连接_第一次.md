# 第一次挥手

![img](/img/372.png)

## FIN 报文

四次挥手第一步，由**主动关闭方**发起。内核生成携带 `FIN=1` 的报文，代表：**本方不会再发送任何新业务数据，单向输出通道关闭。**

## 触发时机

必须满足前置条件：

1. 应用主动执行关闭；

2. 内核发送缓冲区所有数据全部发送完成、收到对应 ACK；

   只有无待确认数据，内核才会发出 FIN 报文，不会丢弃缓存数据。

## 报文

![image-20260616154136254](/img/373.png)

### 源 / 目的端口

```plaintext
Source Port: 3000
Destination Port: 60366
```

- `Source Port: 3000`：服务端主动关闭方端口，本次发送 FIN 的一端，是**第一次挥手发起者**。
- `Destination Port: 60366`：被动接收关闭请求的对端端口。

### 序列号（Sequence Number）

```plaintext
Sequence Number: 180    (relative sequence number)
Sequence Number (raw): 96981201
[Next Sequence Number: 181    (relative sequence number)]
```

- `Sequence Number (raw): 96981201`：原始真实绝对序列号，`Wireshark` 默认展示相对序号简化阅读。
- `Sequence Number: 180（相对序列号）`：本次 FIN 报文的起始序号；TCP 协议规定：**FIN 标志单独占用 1 个序列号**，和数据字节一样消耗序号资源。
- `Next Sequence Number: 181`：下一个将要发送的序号 = 当前 Seq + 1，正好印证 FIN 消耗 1 个序号；语义：主动方已经把 `0~179` 所有业务数据全部发送完毕，用序号 180 代表关闭本方发送通道，之后不再发送任何新数据。

### 确认号（Acknowledgment Number）

```plaintext
Acknowledgment Number: 1135    (relative ack number)
Acknowledgment number (raw): 830817877
```

- `Acknowledgment Number: 1135    (relative ack number)`: **捎带确认作用**，主动方告知对端，**对方发送的 0~1134 所有字节我已经全部完整接收**。

::: tip 为什么第一次挥手会带 ACK？

TCP 支持**捎带确认**机制，不需要单独发一条纯 ACK 报文；

发送 FIN 关闭通道时，可以顺带确认之前收到的对方数据，合并成一条报文，节省网络包，协议完全合法，不改变四次挥手流程。

:::

### 头部长度（Header Length）

```plaintext
0101 .... = Header Length: 20 bytes (5)
```

- `Header Length: 20 bytes (5)`：TCP 头部总长度为 20 字节
- 结构：`20 字节固定头部`。无任何 TCP 扩展选项。

### 标志位（Flags: 0x002 (SYN)）

```plaintext
Flags: 0x011 (FIN, ACK)
    000. .... .... = Reserved: Not set
    ...0 .... .... = Accurate ECN: Not set
    .... 0... .... = Congestion Window Reduced: Not set
    .... .0.. .... = ECN-Echo: Not set
    .... ..0. .... = Urgent: Not set
    .... ...1 .... = Acknowledgment: Set
    .... .... 0... = Push: Not set
    .... .... .0.. = Reset: Not set
    .... .... ..0. = Syn: Not set
    .... .... ...1 = Fin: Set
    [TCP Flags: ·······A···F]
```

- **FIN = Set（1）**

  第一次挥手的核心标识，语义：**我方发送通道关闭，不会再产生新的业务数据发给你**；

- **ACK = Set（1）**

  确认标志置 1，对应上面的 ACK=1135，确认对端历史数据。

### 窗口大小（Window）

```plaintext
Window: 251
[Calculated window size: 64256]
[Window size scaling factor: 256]
```

- `Window: 251`: TCP 头部 16 位原始窗口数值
- `Calculated window size: 64256`：实际接收缓冲区空闲大小 = 251 * 256 = 64256 字节。
- `Window size scaling factor: 256`：三次握手协商的窗口缩放因子

### 校验和（Checksum）

```plaintext
Checksum: 0x68dd [unverified]
[Checksum Status: Unverified]
```

- TCP 校验和，用来校验传输过程中比特差错；
- `unverified` 是因为网卡开启了 **TCP 校验和卸载（Checksum Offload）**，抓包时看到的校验和是硬件计算前的占位值，不是真正的错误。

### 紧急指针（Urgent Pointer）

```plaintext
Urgent Pointer: 0
```

- 因为 `URG=0`，紧急指针字段无效，值为 0。

### TCP 选项

无选项

## 状态变更

发送方发送 FIN 后，主动方从 `ESTABLISHED` 切换至 **FIN_WAIT_1**。

此时内核行为：

1. 不再允许应用层调用 `send()` 写入新数据；
2. 发送缓冲区残留数据会优先全部重传完毕，才会发出 FIN；
3. 开启等待 ACK 的定时器，等待被动方回复确认。

**被动方无变更**
