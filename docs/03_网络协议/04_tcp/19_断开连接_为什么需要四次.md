# 为什么需要四次挥手？

**四次挥手是在 TCP 全双工通信下，双方各自独立关闭发送通道的必要过程。**

## TCP 断开连接到底在做什么？

**TCP 断开连接的核心目的，是双方各自确认"对方不再发送数据"。**

TCP 是全双工协议，两个方向的数据流是**独立**的。断开连接不是"一次性关掉整条连接"，而是每个方向**单独关闭**：

- A → B 方向：A 说"我不再发数据了"，B 确认"好的，我知道了"
- B → A 方向：B 说"我也不再发数据了"，A 确认"好的，我知道了"

每个方向的关闭都需要一次 FIN + 一次 ACK，两个方向加起来就是四次。

## 为什么不能像三次握手那样合并？

### 三次握手为什么只需要三次？

三次握手中，服务端的 **SYN**（同步自己的序列号）和 **ACK**（确认客户端的序列号）可以合并在**同一个报文**中发送：

```plaintext
客户端                          服务端
  |                              |
  |  SYN(seq=x)                  |
  |----------------------------->|
  |                              |
  |  SYN+ACK(seq=y, ack=x+1)     |  ← 两个动作合并成一个报文
  |<-----------------------------|
  |                              |
  |  ACK(seq=x+1, ack=y+1)       |
  |----------------------------->|
```

合并的前提是：**服务端收到 SYN 时，恰好也有 SYN 要发**，两个动作可以同时执行。

### 四次挥手为什么不能合并？

断开连接时，**收到 FIN 的一方可能还有数据没发完**，ACK 和 FIN 无法合并：

```plaintext
客户端                          服务端
  |                              |
  |  FIN(seq=u)                  |  第一次：客户端说"我不发了"
  |----------------------------->|
  |                              |
  |  ACK(ack=u+1)                |  第二次：服务端说"收到，但我还有数据要发"
  |<-----------------------------|
  |                              |
  |          ... 服务端继续发送数据 ...
  |                              |
  |  FIN(seq=w)                  |  第三次：服务端说"我也发完了"
  |<-----------------------------|
  |                              |
  |  ACK(ack=w+1)                |  第四次：客户端说"收到"
  |----------------------------->|
```

**不能合并的根本原因**：第二次挥手（ACK）和第三次挥手（FIN）之间存在**时间差**——服务端可能还有未发送完的业务数据。必须先回 ACK 确认对方的 FIN，等自己的数据全部发完后，才能发出自己的 FIN。

::: tip 特殊情况：四次变三次

如果服务端在收到 FIN 时**恰好也没有数据要发了**，理论上 ACK 和 FIN 可以合并在同一个报文中发送，四次挥手退化为三次。但这在实际中极少发生，因为应用层的关闭操作通常是异步的。

:::

## 为什么需要四次而不是更少？

| 方案 | 问题 |
|------|------|
| 一次挥手 | 双方都无法确认对方是否收到关闭请求 |
| 两次挥手 | 只能关闭一个方向，另一个方向未关闭；无法处理被动方还有数据要发的情况 |
| 三次挥手 | 只有在被动方没有待发送数据时才可行（ACK+FIN 合并），不具有通用性 |
| **四次挥手** | **每个方向独立关闭，保证双方都能完整发送剩余数据后再关闭** |

## 和三次握手的本质区别

| 对比项 | 三次握手（建立连接） | 四次挥手（断开连接） |
|--------|-------------------|-------------------|
| 操作对象 | 双方的初始序列号（ISN） | 双方的结束标记（FIN） |
| 是否可合并 | SYN + ACK 可合并（同时有 SYN 要发） | ACK + FIN 通常不可合并（被动方可能有待发数据） |
| 方向性 | 建立双向通道（对称操作） | 各自关闭单向通道（独立操作） |
| 次数 | 3 次 | 4 次 |
