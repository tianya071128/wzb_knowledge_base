# 为什么需要四次挥手？

核心关键在于一句话：**TCP 是全双工协议，两个方向的数据流必须独立关闭。**

且被动关闭方收到关闭请求时，通常还有未发送完的业务数据，无法将「确认关闭请求」和「发起自身关闭」合并为一条报文，因此必须拆分为四次交互。

## 基础前提：TCP 全双工的特性

TCP 是全双工通信，一条连接里存在**两条完全独立的单向数据流**：

- 方向 1：主动方 → 被动方（发送通道）
- 方向 2：被动方 → 主动方（发送通道）

两条通道的生命周期可以不同步：一方关闭自己的发送通道后，另一方仍然可以继续发送数据，这就是 TCP 的**半关闭（Half-Close）**特性。

关闭一条单向通道，需要一次「FIN 关闭请求 + ACK 确认」，两条通道就对应两组交互，这是四次挥手的底层逻辑基础。

## 为什么不能像建立连接那样合并成 3 次？

建立连接（三次握手）时：

- A 发 `SYN`。
- B 回复 `SYN+ACK`（**合二为一**），因为 B 在收到 A 的请求时，**本身也是空闲的**，立刻就能同步回应。
- A 回复 `ACK`。

但在**断开连接**时：

- B 收到 A 的 `FIN`（第一次挥手），**B 可能还在忙着发送最后一波数据**。
- B 不能立刻也说“我也要断开”，因为他的数据还没传完。他必须先发一个 `ACK`（第二次挥手）告诉 A：“你发完了我知道了，你先等一下，我还在传数据。”
- 等到 B 把数据发完了，他才主动发送 `FIN`（第三次挥手）进行关闭。
- **ACK 和 FIN 在时间上被分开了**，导致了“4 次”。

## 特殊情况：什么时候看起来像「三次挥手」？

如果被动方收到 FIN 时，**本身已经没有任何数据要发送了**，应用层也立刻调用了 close，那么内核理论上可以把「第二次的 ACK」和「第三次的 FIN」合并到同一条报文中发送，此时抓包就会看到 3 条报文：`FIN → FIN+ACK → ACK`。

但这只是报文层面的合并，**逻辑上仍然是两个方向的独立关闭**，本质还是四次挥手的语义，只是做了捎带优化。绝大多数有业务数据传输的场景，仍然是标准的四次报文。
