# 拥塞控制

**拥塞控制**：管控**网络链路负载**，防止网络拥堵、队列溢出、大面积丢包，避免传输瘫痪。

## 为什么要拥塞控制？

网络中的路由器和链路带宽是共享资源。如果所有 TCP 连接都只按照接收方的接受能力（流量控制窗口 `rwnd`）发送数据，大量流量汇聚到某一跳时，就会造成：

- 路由器缓存溢出 → 丢包
- 延迟急剧增加
- 网络吞吐量严重下降（拥塞崩溃）

因此，发送方还需要根据**网络的拥塞程度**动态调整发送速率。这个动态调整的窗口叫做**拥塞窗口（cwnd）**，实际发送窗口为：
`发送窗口 = min(rwnd, cwnd)`

## 拥塞窗口 cwnd

**Congestion Window**，拥塞窗口

代表：**当前网络链路能够平稳承载、不会引发拥塞的最大未确认数据量**

单位：**TCP 报文段个数**

### 作用

1. 防止瞬间海量报文涌入路由队列，造成排队溢出丢包
2. 动态适配网络拥堵程度，网络宽松提速、拥堵降速
3. 平衡带宽利用率与传输稳定性

### 关联阈值：ssthresh

**慢启动阈值**，分界 cwnd 增长模式

- `cwnd < ssthresh`：慢启动阶段，**指数增长**
- `cwnd ≥ ssthresh`：拥塞避免阶段，**线性增长**

初始 ssthresh 一般设为连接允许最大窗口值，拥塞发生后会减半下调。

### 不同阶段 cwnd 变化规则

1. **慢启动阶段**

   初始 cwnd 取值很小，主流规范初始为 2~4 个报文段

   规则：每收到**一个报文段的 ACK**，`cwnd += MSS`

   一轮 RTT 往返后，cwnd 整体翻倍，指数上涨

   **示例**

   初始 cwnd=2

   一轮往返 → 4 → 8 → 16，触达 ssthresh 后切换模式

2. **拥塞避免阶段**

   不再激进翻倍

   规则：**一个 RTT 周期结束**，`cwnd += MSS * (MSS / cwnd) `，大约增加 1 个 MSS，匀速缓慢抬升

   避免带宽抢占过猛，平稳占用链路资源

3. **轻微丢包：3 个重复 ACK 触发快重传**

   判定：网络轻微抖动、局部丢包，非全网严重拥堵

   cwnd 变更流程

   1. `ssthresh = cwnd / 2`
   2. `cwnd = ssthresh`
   3. 后续直接进入拥塞避免线性增长

   特点：窗口只腰斩、不归零，网速平滑回落

4. **严重拥塞：超时重传触发**

   判定：链路堵塞严重、大面积报文丢失

   cwnd 变更流程

   1. `ssthresh = cwnd / 2`
   2. cwnd 重置为初始最小值
   3. 重新从慢启动开始试探提速

   特点：窗口断崖式下跌，重新低速探测网络

## 流量控制 vs 拥塞控制

| 对比 | 流量控制       | 拥塞控制               |
| ---- | -------------- | ---------------------- |
| 范围 | 端到端主机     | 全网链路路由           |
| 窗口 | rwnd 接收窗口  | cwnd 拥塞窗口          |
| 诱因 | 接收缓存爆满   | 网络拥堵丢包、时延暴增 |
| 目的 | 不把接收方撑爆 | 不把网络堵死           |

## 四大核心阶段

TCP 拥塞控制四阶段：**慢启动 → 拥塞避免 → 快重传 → 快恢复**

### 慢启动 (Slow Start)

- **触发条件**：连接刚建立、[RTO 超时](/tcp/data_retransmission.html#rto-重传超时时间-的计算)后重新开始。
- **目标**：快速探测网络可用带宽，而不是真的“慢”。
- **操作**：
  - 初始 cwnd 通常为 10 MSS (RFC 6928)。
  - 每收到一个**新的 ACK**，`cwnd += MSS`。
  - 这导致每经过一个 RTT，cwnd 翻倍（指数增长）。
- **离开条件**：
  - cwnd ≥ ssthresh → 进入**拥塞避免**阶段。
  - 发生丢包 → 调整 ssthresh，cwnd 减半或降到 1 MSS，并进入相应恢复。

### 拥塞避免 (Congestion Avoidance)

- **触发条件**：cwnd 达到或超过 ssthresh，且网络未出现丢包。
- **目标**：温和探测带宽，避免引起剧烈拥塞。
- **操作** (AIMD 的“加性增大”部分)：
  - 每收到一个**新的 ACK**，`cwnd += MSS * (MSS / cwnd)`。
  - 这样每个 RTT 大约增加 1 个 MSS，变为**线性增长**。
- **丢包处理**：
  - **超时 (RTO)**：`ssthresh = max(2*MSS, cwnd/2)；cwnd = 1 MSS`；重新进入慢启动。
  - **连续 3 个重复 ACK**：进入快速重传和快速恢复。

### 快速重传 (Fast Retransmit)

- **原理**：收到 3 个或以上重复 ACK 时，意味着后续报文大概率已丢失（但仍能收到部分包，网络未完全瘫痪）。此时不等超时重传定时器过期，立刻重传丢失的报文。
- **触发条件**：同一个报文被 ACK 过 4 次（首次 ACK + 3 个重复 ACK）。
- **后续动作**：不将 cwnd 降为 1，而是进入**快速恢复**。

### 快速恢复 (Fast Recovery)

- **目标**：利用收到的重复 ACK 表明网络还能交付部分数据，避免重新慢启动造成吞吐量骤降。
- **Reno 的操作步骤**：
  1. ssthresh = cwnd / 2 （乘性减小）。
  2. cwnd = ssthresh + 3 * MSS （加上 3 个重复 ACK 已经代表离开网络的报文）。
  3. 重传丢失报文。
  4. 此后每收到**一个重复 ACK**（说明又有一个包离队）：`cwnd += MSS`（临时膨胀窗口，以便能发送新数据）。
  5. 当收到**新的 ACK**（确认了新数据，包括重传的包）：
     `cwnd = ssthresh`（撤销膨胀部分，回到拥塞避免阶段的初始值），然后进入**拥塞避免**。
- **局限性 (Reno)**：如果在一个窗口中丢失了多个报文，Reno 可能每次只能通过“部分 ACK”发现一个丢失，导致反复快速重传/恢复，甚至降到 ssthresh 以下而超时。**NewReno** 改进了这一点：在快速恢复期间，通过部分 ACK 判断后续丢包并直接重传，直到所有已丢失的包都被恢复才退出快速恢复，并将 cwnd 设为 ssthresh。

## 拥塞信号的响应对比

| 事件      | RTO 超时                         | 3 个重复 ACK             |
| :-------- | :------------------------------- | :----------------------- |
| 严重程度  | 网络可能极度拥塞，几乎没有包送达 | 偶尔丢包，后续包仍可到达 |
| cwnd 调整 | 降到 1 MSS，重新慢启动           | 降到一半（快速恢复）     |
| ssthresh  | 设为当前 cwnd/2                  | 设为当前 cwnd/2          |
| 进入状态  | 慢启动                           | 快速恢复后进入拥塞避免   |

## 现代 TCP 拥塞控制算法演进

传统的 Reno 在高带宽、高延迟网络（BDP 较大）中，拥塞避免每 RTT 只增加 1 MSS，恢复带宽极慢，且公平性差。现代算法针对这些问题做了大量优化。
