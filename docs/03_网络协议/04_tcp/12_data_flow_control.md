# 流量控制

流量控制是**端到端**的限速机制，作用：**防止发送方发送速率过快，超出接收方缓冲区处理能力，造成数据丢失**。

## 实现原理: 接收窗口 rwnd

**rwnd = Receive Window**，接收窗口。**接收方 TCP 缓冲区当前剩余空闲字节数**，用来告诉发送方：你最多还能一次性发给我多少数据。

### 窗口数值协商

两端的每个 TCP 报文头部，固定 16 位 [ **Window Size** 字段](/tcp/header_format.html#窗口大小-16-位), 用于告诉对方还能接收多少字节的数据

::: tip 窗口缩放 选项

原始 TCP 报文头部的 [ **Window Size** 字段](/tcp/header_format.html#窗口大小-16-位), 最大只有 `64KB`, 已无法满足现代化网络。

[窗口缩放（Window Scale）](/tcp/header_option.html#窗口缩放-window-scale) 选项将 16 位窗口大小字段的表示范围从 64KB 扩展到最大 **1GB**

:::

### 约束规则

发送方允许发送的未确认数据总量，**不能超过对方通告的 rwnd**

公式: 待发数据量 ≤ 对端接收窗口 rwnd

### 窗口数值动态变化

1. 发送方发数据 → 接收方的缓冲区被占用 → **rwnd 变小**
2. 接收方的应用层调用 read 读走数据 → 接收方的缓冲区腾出空间 → **rwnd 变大**
3. 接收方每次 ACK 携带最新 rwnd，发送方实时更新发送上限

## 与拥塞窗口 cwnd 的关系

- **接收窗口 rwnd**：决定本机能不能收下数据，流量控制使用
- **拥塞窗口 cwnd**：决定网络能不能承载数据，拥塞控制使用
- 实际最大发送量：`send_max = min(rwnd, cwnd)`

## 工作流程

1. 连接建立，双方初始化窗口大小
2. 发送方依据对方窗口大小，批量发送数据
3. 数据存入接收缓冲区，占用缓存空间
4. 接收方回复 ACK 时，同步上报**缩小后的剩余窗口**
5. 发送方识别窗口变小，主动降低发送速率
6. 接收方的应用层调用`read`读取数据，缓冲区释放空闲空间
7. 下次应答携带更大窗口值，发送方恢复提速

```plaintext
发送方                  接收方
   |                       |
   |  发送300字节数据       |
   |----------------------->|
   |                       |  缓冲区占用300，剩余200
   |                       |
   |       ACK + rwnd=200  |
   |<-----------------------|
   |  限制最多再发200字节    |
   |                       |
   |  发送200字节数据       |
   |----------------------->|
   |                       |  缓冲区占满，rwnd变为0
   |                       |
   |       ACK + rwnd=0    |
   |<-----------------------|
   |  立刻暂停发送新数据     |
   |                       |
   |                       |  应用层读取400字节
   |                       |  缓冲区空余400
   |       ACK + rwnd=400  |
   |<-----------------------|
   |  恢复正常发送数据      |
   |----------------------->|
```

## 窗口关闭

接收方缓冲区被数据占满，通告**窗口值 rwnd=0**，发送方停止新发数据，进入暂停传输状态，即为窗口关闭。

### 触发原因

1. 发送速率过快，数据持续涌入接收缓冲区
2. 应用层迟迟不调用`read`读取内核缓存数据
3. 缓冲区满载，无空闲空间接纳新报文

### 交互流程

```plaintext
发送方                接收方
|                     |
|  先发 200字节       |
|-------------------> |
|                     | 缓存占用200，剩余200
|    ACK + rwnd=200   |
|<------------------- |
|                     |
|  再发200字节        |
|-------------------> |
|                     | 缓存彻底占满，无空余
|    ACK + rwnd=0     | 窗口正式关闭
|<------------------- |
|  收到0窗口，停止发送 | 暂停所有新数据发包
|                     |
```

### 窗口扩大的 ACK 报文丢失

当窗口关闭后，接收方处理完数据后，会向发送方通告一个窗口非 0 的 ACK 报文，如果这个通告窗口的 ACK 报文(**ACK 报文不会重传**)在网络中丢失了的话：

#### 零窗口探测（核心兜底）

为了解决这个问题，TCP 为每个连接设有一个持续定时器，**只要 TCP 连接一方收到对方的零窗口通知，就启动持续计时器**。

周期性发送**1 字节探测报文**，查询对方窗口状态

- 探测收到 `rwnd=0`：继续等待，下次再探
- 探测收到 `rwnd>0`：立即恢复数据发送
- 探测未回复, 按间隔延后再次发起探测。多次未回复, 
- 达到阈值后，判定对方主机宕机、链路断开，**主动关闭 TCP 连接**，释放套接字资源。

## 糊涂窗口综合症

如果接收方腾出几个字节并告诉发送方现在有几个字节的窗口，就会反复产生**极小窗口**和**极小数据包**，导致**头部占比极高、带宽利用率极低、传输效率极差**的慢性病。

### 现象与危害

**典型表现**

- 每次发送**1 字节数据**，却带 **20B TCP 头 + 20B IP 头 = 40B 头部**
- 数据包总长度 **41B**，有效载荷仅 **1B**，**效率 ≈ 2.4%**
- 网络里塞满大量**小包**，设备转发压力大、吞吐量上不去

**和 “零窗口” 的区别**

- **零窗口（rwnd=0）**：急症，**完全停发**，连接僵住
- **糊涂窗口**：慢性病，**一直发但全是小包**，慢且低效

### 原因

1. 发送方：太 “勤快”，小数据立即发
   * 应用**逐字节产生数据**（如 Telnet/SSH 键盘输入）
   * TCP 不等待、不合并，**来 1 字节就发 1 字节**
   * 结果：海量 **41B 小包**

2. 接收方：太 “小气”，小空间就通告
   * 应用**读数据极慢**，接收缓冲区**只腾出一点点空间**
   * 接收方**立刻通告 rwnd=1/2/3...**，引诱发送方发小包
   * 恶性循环：**小窗口 → 小包 → 小窗口 → 小包**

### 解决方案

**发送方 + 接收方 双端配合**

1. **发送方：Nagle 算法（RFC 896）——“攒够再发”**
    * 有**未确认（in-flight）数据**时，**新小数据先缓存，不发**
    * 直到：
      - 收到**之前数据的 ACK**；**或**
      - 新数据**攒够 1 个 MSS**；**或**
      - 无 in-flight 数据（可立即发）
    * 效果：**把多个1B 小包合并成1 个 MSS 大包**，大幅减少包数量、提升效率。
2. **接收方：Clark 方案 + 延迟确认 ——“够大再通告”**

    1. **Clark 规则（RFC 813）不主动通告小窗口**，满足任一才更新 rwnd：

       * 空闲空间 ≥ **1 个 MSS**；**或**
       * 空闲空间 ≥ **接收缓冲区的一半**

       否则：**隐瞒小窗口，继续通告 rwnd=0**，避免引诱小包。

    2. **延迟确认（Delayed ACK）**

       * 收到数据**不立即 ACK**，等待 **200ms（默认）** 或**攒够数据**再确认
       * 目的：
         - 减少 ACK 包数量；
         - 让应用有时间**多读点数据**，一次性腾出大空间，**直接通告大窗口**
