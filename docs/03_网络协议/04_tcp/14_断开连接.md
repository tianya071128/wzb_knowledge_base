# 断开连接

TCP 连接断开的过程，通常就是我们常说的“四次挥手”。因为 TCP 是全双工通信，每个方向都需要独立关闭，所以需要**四次交互**。

## 四次挥手

TCP 是双工通信, 双方都可以主动断开连接, 四次挥手的过程如下图：

![img](/img/372.png)

| 步骤 | 发送方          | 报文                                     | 状态变化（主动方 / 被动方）                  |
| :--- | :-------------- | :--------------------------------------- | :------------------------------------------- |
| 1    | 主动方 → 被动方 | `FIN=1, seq=x`                           | `FIN_WAIT_1`                                 |
| 2    | 被动方 → 主动方 | `ACK=1, ack=x+1`                         | 主动方：`FIN_WAIT_2` 被动方：`CLOSE_WAIT`    |
| 3    | 被动方 → 主动方 | `FIN=1, seq=y, ACK=1, ack=x+1`（可捎带） | 被动方：`LAST_ACK`                           |
| 4    | 主动方 → 被动方 | `ACK=1, seq=x+1, ack=y+1`                | 主动方：`TIME_WAIT` 被动方收到后进入`CLOSED` |

## 异常断开

并非所有断开都是四次挥手，主要分为五大类：**RST 主动复位、超时断开、半开连接、中间设备切断、应用层强制关闭**。

### 对端直接发送 RST 报文

TCP 头部 `RST=1`，**无需应答、不排队、不等待缓冲区数据**。

收到 RST 的一方，连接立即置为 `CLOSED`，无 `TIME_WAIT` / `CLOSE_WAIT` 状态。

**常见触发场景：**

* 访问未监听端口：客户端向目标 IP 的**未开放端口**发起 TCP 连接
  * 客户端发 `SYN` 握手
  * 服务端内核发现该端口无进程监听，**直接回复 RST**
  * 连接建立失败，立刻断开
* 收到非法 / 异常报文：已建立的连接中，收到格式错误、序列号越界、超出窗口的非法报文
  * 报文序列号不在对方接收窗口范围内
  * 畸形 TCP 头、协议字段非法
  * 内核判定连接异常，主动发 RST 断开。
* 服务端进程崩溃 / 强制杀死：服务端应用进程（Nginx、Java 服务等）被 `kill -9`、闪退、崩溃
  * 进程退出，套接字被内核回收
  * 内核向对端发送 **RST**，强制断连
  * 客户端读写 `recv/send` 直接返回错误码（如 `Connection reset by peer`）

### 超时类断开（无 RST、无 FIN，静默断连）

双方无任何控制报文交互，依靠内核定时器判定连接死亡，分为两类定时器。

1. **重传超时（RTO 超时）**： 

   **触发场景**：网络完全中断、路由故障、对端宕机断网，**报文发出去后永久收不到 ACK**

   1. 发送方发送数据，启动 RTO 计时器
   2. 超时未收到 ACK，执行重传，且 RTO 指数退避
   3. 达到系统配置**最大重传次数**后，内核判定链路彻底失效
   4. **静默断开连接**（部分系统会补发 RST，部分直接标记关闭）

   **典型表现**

   - 抓包：只有不断重传的 TCP 数据段，无任何回应
   - 应用报错：`Connection timed out`
   - 区别：和 RST 报错文案不一样。

2. **TCP 保活超时（Keepalive 超时）**

   **触发场景**：连接长期空闲（无数据收发），双方网络看似通，但对端主机宕机、拔网线、休眠，**无任何报文响应**。

   1. 连接空闲达到 `tcp_keepalive_time`，内核发送**保活探测包**
   2. 多次探测（`tcp_keepalive_probes`）全部无应答
   3. 内核判定连接为僵死连接，主动断开（通常伴随发送 RST）

   **适用场景**

   - 长时间空闲的长连接（数据库连接、SSH、代理隧道）
   - 断网、设备休眠、内网机器断电

### 半开连接（Half-Open Connection）断开

**一方意外断网 / 断电 / 宕机，未发送 FIN/RST**，另一方连接状态仍显示 `ESTABLISHED`，形成 “假活连接”。

- 断网侧：连接已消失，但没发任何关闭报文
- 正常侧：内核仍认为连接正常，持续等待数据 / ACK

只能依靠 **TCP 保活定时器** 或 **应用层心跳**：

1. 开启系统 TCP 保活：探测无响应 → 发 RST 断连
2. 应用层自定义心跳：心跳超时 → 应用主动 `close()` 或发 RST
3. 主动发业务数据：触发重传超时 → 连接断开

**典型场景**

- 电脑 / 手机休眠、拔网线、断电
- 跨网设备意外下线，无正常挥手

### 中间网络设备强制切断（防火墙 / 网关 / 负载均衡）

链路两端主机正常，但**中间网络设备**主动掐断连接，是线上环境非常常见的异常断开。

### 应用层强制暴力关闭

应用程序不遵循优雅关闭逻辑，直接强制销毁套接字：

1. **直接 close 丢弃缓冲区数据**

   部分编程语言 / 框架，调用 `close()` 时，**不等待内核发送缓冲区数据发完**，直接释放套接字：

   * 内核会把缓冲区未发送数据全部丢弃
   * 同时向对端发送 RST，而非正常 FIN

2. **setsockopt 强制断连**

   通过 Socket 选项设置 `SO_LINGER` 为强制关闭：

   ```plaintext
   SO_LINGER 开启 +  linger_time = 0
   ```

   效果：调用 `close` 时**立刻释放套接字，不排队、不发剩余数据，直接发 RST**。

   多用于紧急释放连接、高并发场景快速回收资源，但会造成数据丢失。

3. **进程异常退出**

   进程被强杀、崩溃、OOM 杀死，内核回收 fd 并发送 RST。

### 对比表

| 断开类型                 | 是否有 RST | 有无四次挥手 | 典型报错 / 现象              | 核心成因                    |
| ------------------------ | ---------- | ------------ | ---------------------------- | --------------------------- |
| 对方进程崩溃 / 强杀      | 有         | 无           | Connection reset by peer     | 对端 fd 被内核回收          |
| 访问未监听端口           | 有         | 无           | 连接被拒绝                   | 目标端口无监听进程          |
| 重传超时                 | 部分有     | 无           | Connection timed out         | 网络中断、报文无应答        |
| TCP 保活超时（半开连接） | 通常有     | 无           | 连接卡死，最终超时断开       | 对端断网 / 断电，无任何报文 |
| 防火墙 / NAT 空闲超时    | 部分有     | 无           | 长连接空闲一段时间后突然断连 | 中间设备回收连接表          |
| SO_LINGER 强制关闭       | 有         | 无           | 数据丢失，连接立即断开       | 应用主动暴力释放套接字      |
