# 第四次挥手

![img](/img/372.png)

## 基础定位

**发起方**：主动关闭方

**报文类型**：ACK 确认报文

**核心作用**：主动方确认收到对方的关闭通知，完成双向关闭的最后一步确认，连接即将彻底释放

## 触发时机

**第四次挥手和第二次挥手的触发逻辑完全一致：**由操作系统内核自动触发，无需应用层参与**。**

完整触发时序：

1. 主动方内核收到第三次挥手的 FIN 报文，识别到对方也关闭了发送通道。
2. 内核立刻构造 ACK 确认报文，无需等待、不走延迟确认机制，即时回复。
3. 发送 ACK 的同时，内核自动将连接状态切换为 TIME_WAIT，启动 2MSL 计时器。

## TIME_WAIT：第四次挥手的“附加产物”

第四次挥手发送后，主动方为什么会进入 `TIME_WAIT` 而不是直接关闭？这是 TCP 可靠性设计的高潮，有两个核心目的：

### 1. 确保最后一个 ACK 能被对方收到

如果第四次挥手的 ACK 在网络中丢失，被动方（处于 `LAST_ACK`）收不到确认，会重传它的 FIN（第三次挥手）。
主动方在 `TIME_WAIT` 期间可以：

- 重新收到重传的 FIN
- 再次发送 ACK（重置 2MSL 计时器）

这样**保证了双方都能可靠地确认连接关闭**。如果主动方直接关闭，那么 ACK 丢失后，被动方永远无法正常关闭，会一直处于 `LAST_ACK` 不断重传 FIN 直至超时。

### 2. 让旧连接的所有报文“消散”在网络中

一个连接由四元组（源IP、源端口、目的IP、目的端口）标识。如果主动方立即关闭并立即用同样的四元组建立新连接，网络上可能还有残留的旧连接数据报文延迟到达，会被误认为是新连接的合法数据，造成数据错乱。
等待 2MSL 就是为了保证旧连接**两个方向**上所有延迟或重传的报文都已在网络中消失，避免干扰后续同四元组的连接。

### 补充拓展问题

1. 为什么是 2 倍 MSL，不是 1 倍？

   * 1MSL：保证旧连接所有单向报文全部消失；
   * 再加 1MSL：保证对方针对旧报文回复的响应包也全部过期丢弃；只等 1MSL 无法覆盖往返完整周期，残留应答包仍会干扰新连接。

2. 为什么被动关闭方不需要 TIME_WAIT？

   被动方收到第四次 ACK 直接 CLOSED：

   * 被动方不会发送任何收尾报文，不存在重传 FIN 的需求；
   * 就算旧延迟报文到达，被动方早已销毁该五元组连接，只会丢弃报文，不会产生连锁异常。

3. 线上大量 TIME_WAIT 会有什么影响？

   主动端端口被占用 2MSL，短时间无法复用，高并发短连接场景会出现端口耗尽、无法新建连接。

## 报文

![image-20260616172955112](/img/376.png)

### 源 / 目的端口

```plaintext
Source Port: 3000
Destination Port: 60366
```

- `Source Port: 3000`：**主动发起关闭的端口**，收到第三次挥手 FIN 后，内核自动生成这条 ACK
- `Destination Port: 60366`：被动关闭方端口，收到本条 ACK 后，连接直接切换为 `CLOSED`，无需等待。

### 序列号（Sequence Number）

```plaintext
Sequence Number: 181（相对序列号）
Sequence Number (raw): 96981202
[Next Sequence Number: 181]
```

- `Sequence Number: 181（相对序列号）`：第一次挥手 `Seq=180` 携带 FIN，消耗 1 个序号，因此主动方后续基准序号固定为 181

- `Sequence Number (raw): 96981202`：原始真实绝对序列号，Wireshark 默认展示相对序号简化阅读。

- `[Next Sequence Number: 181]`：本条是**纯 ACK 报文**，没有数据、没有 FIN，**不消耗序列号**；

### 确认号（Acknowledgment Number）

```plaintext
Acknowledgment Number: 1136（相对 ACK）
Acknowledgment number (raw): 830817878
```

- `Acknowledgment Number: 1136（相对 ACK）`: 第三次挥手报文的 Seq 是 `1135`，并且携带 FIN 标志；**FIN 会占用 1 个序列号**，确认号必须在对方 Seq 基础上 +1；

### 头部长度（Header Length）

```plaintext
0101 .... = Header Length: 20 bytes (5)
```

- `Header Length: 20 bytes (5)`：TCP 头部总长度为 20 字节
- 结构：`20 字节固定头部`。无任何 TCP 扩展选项。

### 标志位（Flags: 0x002 (SYN)）

```plaintext
Flags: 0x010 (ACK)
    000. .... .... = Reserved: Not set
    ...0 .... .... = Accurate ECN: Not set
    .... 0... .... = Congestion Window Reduced: Not set
    .... .0.. .... = ECN-Echo: Not set
    .... ..0. .... = Urgent: Not set
    .... ...1 .... = Acknowledgment: Set
    .... .... 0... = Push: Not set
    .... .... .0.. = Reset: Not set
    .... .... ..0. = Syn: Not set
    .... .... ...0 = Fin: Not set
    [TCP Flags: ·······A····]
```

- **ACK = Set（1）**

  唯一生效标志，代表 ACK=1136 字段有效，是对第三次挥手 FIN 的正式确认；

### 窗口大小（Window）

```plaintext
Window: 251
[Calculated window size: 64256]
[Window size scaling factor: 256]
```

- `Window: 251`: TCP 头部 16 位原始窗口数值
- `Calculated window size: 65280`：实际接收缓冲区空闲大小 = 251 * 256 = 64256 字节。
- `Window size scaling factor: 256`：三次握手协商的窗口缩放因子

### 校验和（Checksum）

```plaintext
Checksum: 0x68dc [unverified]
[Checksum Status: Unverified]
```

- TCP 校验和，用来校验传输过程中比特差错；
- `unverified` 是因为网卡开启了 **TCP 校验和卸载（Checksum Offload）**，抓包时看到的校验和是硬件计算前的占位值，不是真正的错误。

### 紧急指针（Urgent Pointer）

```plaintext
Urgent Pointer: 0
```

- 因为 `URG=0`，紧急指针字段无效，值为 0。

### TCP 选项

无选项

## 状态变更

### 主动关闭方

发送完本次 ACK 报文后，连接状态从 `FIN_WAIT_2` 切换为 `TIME_WAIT`

- 必须等待固定时长 **2MSL**（报文最大生存时间的 2 倍）后，才会进入 CLOSED 状态、彻底释放连接。

### 被动关闭方

收到本次 ACK 报文后，连接状态从 `LAST_ACK` 直接切换为 `CLOSED`

- 无需额外等待，收到最终确认后立刻释放连接所有资源，连接彻底关闭。
- 这也是主动方才有 TIME_WAIT、被动方没有的原因：被动方收到确认就可以直接收尾。
