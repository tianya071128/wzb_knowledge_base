# 挥手中丢失包

## 第一次挥手丢失包

主动关闭方发送 FIN 后，进入 `FIN_WAIT_1` 状态，等待被动方的 ACK 确认。

如果 FIN 丢失，被动方完全不知道对方要关闭连接，不会有任何响应。

**主动方行为**：收不到 ACK，触发 FIN 超时重传。

重传机制：

1. 超时时间：Linux 下由 `tcp_orphan_retries` 内核参数控制，默认 7 次
2. 采用**指数退避**重试：间隔时间是上一次的 2 倍
3. 超过最大重传次数后，连接强制关闭，主动方退回 `CLOSED` 状态

::: tip 注意

**重传 FIN 报文，序列号不变**

和重传 SYN 一样，FIN 的序列号是固定的，重传只是重发同一个报文。

:::

**被动方行为**：无任何变化，连接仍然处于 `ESTABLISHED` 状态，可以正常发送数据。

## 第二次挥手丢失包

主动方发送 FIN 后进入 `FIN_WAIT_1`，被动方收到 FIN 后进入 `CLOSE_WAIT` 并回复 ACK。

如果这个 ACK 丢失：

此时双方状态：

- **主动方**：停留在 **FIN_WAIT_1**，等待 ACK
- **被动方**：进入 **CLOSE_WAIT**，认为已经确认了对方的关闭请求

**两端各自行为**：

- **主动方**：收不到 ACK，不确定 FIN 是否到达 → 超时重传 **FIN 报文**
- **被动方**：已进入 `CLOSE_WAIT`，不受影响，可以继续发送剩余数据

**自动修复机制**：

1. 主动方重传 FIN
2. 被动方收到重复 FIN，重新回复 ACK
3. 主动方收到 ACK，进入 `FIN_WAIT_2`
4. 流程恢复正常

::: tip 注意

被动方收到重复的 FIN 不会报错，TCP 协议规定：**重复的 FIN 必须被确认**。这是因为被动方的 ACK 可能丢失，主动方只能通过重传 FIN 来确保对方收到了关闭请求。

:::

## 第三次挥手丢失包

被动方发送完所有剩余数据后，发送 FIN 进入 `LAST_ACK` 状态。

如果这个 FIN 丢失：

此时双方状态：

- **主动方**：停留在 **FIN_WAIT_2**，等待被动方的 FIN
- **被动方**：进入 **LAST_ACK**，等待主动方的最终 ACK

**两端各自行为**：

- **主动方**：无法区分 FIN 丢失还是被动方还有数据要发，只能继续等待。如果等待超时（`tcp_fin_timeout`，默认 60 秒），连接强制关闭
- **被动方**：收不到 ACK，触发 FIN 超时重传

**自动修复机制**：

1. 被动方重传 FIN
2. 主动方收到 FIN，回复 ACK
3. 被动方收到 ACK，进入 `CLOSED`
4. 流程恢复正常

## 第四次挥手丢失包

主动方收到被动方的 FIN 后，回复 ACK 并进入 `TIME_WAIT` 状态。

如果这个 ACK 丢失：

此时双方状态：

- **主动方**：进入 **TIME_WAIT**，开始 2MSL 倒计时
- **被动方**：停留在 **LAST_ACK**，等待 ACK

**两端各自行为**：

- **主动方**：已进入 `TIME_WAIT`，2MSL 期间如果收到被动方重传的 FIN，会重新回复 ACK 并重置 2MSL 计时器
- **被动方**：收不到 ACK，触发 FIN 超时重传

**自动修复机制**：

1. 被动方重传 FIN
2. 主动方在 `TIME_WAIT` 期间收到 FIN，重新回复 ACK，重置 2MSL 计时器
3. 被动方收到 ACK，进入 `CLOSED`
4. 主动方等待 2MSL 结束后，进入 `CLOSED`

::: tip 这就是 TIME_WAIT 存在的核心理由之一

如果主动方发送完 ACK 后直接进入 `CLOSED`，被动方重传的 FIN 将无人应答（收到 RST），导致被动方无法优雅关闭。`TIME_WAIT` 确保主动方在 2MSL 内仍然"活着"，能够处理被动方的 FIN 重传。

:::

## 总览

| 丢失的报文 | 主动方状态 | 被动方状态 | 修复方式 |
|-----------|-----------|-----------|---------|
| 第一次 FIN | `FIN_WAIT_1` | `ESTABLISHED` | 主动方重传 FIN |
| 第二次 ACK | `FIN_WAIT_1` | `CLOSE_WAIT` | 主动方重传 FIN，被动方重新回复 ACK |
| 第三次 FIN | `FIN_WAIT_2` | `LAST_ACK` | 被动方重传 FIN |
| 第四次 ACK | `TIME_WAIT` | `LAST_ACK` | 被动方重传 FIN，主动方重新回复 ACK |

**核心规律**：

- **ACK 丢失**：不会重传 ACK 本身，而是由对方重传对应的 FIN
- **FIN 丢失**：发送方超时重传 FIN，序列号不变
